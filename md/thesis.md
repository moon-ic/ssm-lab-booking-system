# 基于SSM框架的高校实验室设备预约管理系统设计与实现

## 第2章 相关技术介绍

### 2.1 SSM框架概述

SSM框架是Java Web开发中应用较为广泛的技术组合，通常由Spring、Spring MVC和MyBatis三部分组成。其中，Spring主要负责对象管理与业务层解耦，Spring MVC负责请求分发与Web层控制，MyBatis负责关系型数据库访问与数据持久化。三者协同工作，能够构建层次清晰、职责明确的企业级应用架构。

本系统在整体架构设计上采用SSM思想，并以Spring Boot为基础进行工程化实现。相较于传统SSM项目中较为繁琐的XML配置方式，Spring Boot通过自动配置机制简化了项目初始化与运行部署过程，同时仍保留了Spring生态下的典型分层模式。系统后端主要划分为控制层、业务层和数据访问层：控制层负责接收与响应客户端请求，业务层负责处理系统核心逻辑，数据访问层负责与数据库进行交互。通过这种分层方式，系统各功能模块之间的耦合度得以降低，从而提高了系统的可维护性与可扩展性。

### 2.2 Spring框架

Spring是SSM体系中的核心框架，其主要优势在于控制反转（Inversion of Control，IoC）和依赖注入（Dependency Injection，DI）机制。传统开发模式下，对象的创建与依赖关系通常由程序员在代码中直接管理，容易导致模块之间耦合紧密；而Spring通过IoC容器统一管理对象生命周期，将对象依赖关系的维护交由容器完成，从而增强了系统的灵活性与可维护性。

除IoC与DI外，Spring还提供了面向切面编程（Aspect Oriented Programming，AOP）支持，可将事务控制、日志记录、权限校验等横切关注点从业务逻辑中分离出来，以声明式方式织入目标方法。该机制有助于减少重复代码，提高系统开发效率。

在本系统中，Spring主要承担以下作用：一是统一管理控制层、业务层和持久层相关组件；二是协调各层之间的依赖关系；三是为事务控制和后续功能扩展提供基础支撑。借助Spring框架，系统整体结构更加清晰，模块之间的职责划分也更加明确。

### 2.3 Spring MVC

Spring MVC是Spring体系中用于构建Web应用的重要模块，其核心思想是基于模型—视图—控制器（Model-View-Controller，MVC）模式实现请求处理流程。该框架通过前端控制器统一接收客户端请求，并根据映射关系分发至相应的控制器方法进行处理，随后将处理结果返回给客户端。

在本系统中，Spring MVC主要用于实现前后端之间的数据交互。前端通过HTTP请求访问系统功能接口，后端控制器接收请求参数并调用业务层完成相应处理，最终将结果以JSON格式返回。由于系统采用前后端分离架构，因此Spring MVC在其中主要承担接口组织、参数绑定、数据校验以及异常响应等职责。

基于Spring MVC构建的Web层具有良好的扩展性与统一性，有利于实现接口风格规范化，并提高系统的开发效率和后续维护能力。

### 2.4 MyBatis框架

MyBatis是一种轻量级持久层框架，能够实现Java对象与关系型数据库之间的数据映射。与全自动ORM框架相比，MyBatis允许开发者直接编写SQL语句，对查询逻辑和执行过程具有更高的可控性，因此较适合业务规则较明确、对SQL性能与可读性要求较高的应用场景。

本系统采用MyBatis作为数据持久化实现框架，通过Mapper接口与映射文件相结合的方式完成数据库访问操作。对于用户、设备、预约、借用、维修等核心业务数据，系统均通过MyBatis完成增删改查操作，并借助结果映射机制实现数据库字段与实体属性之间的对应转换。该方式既保留了SQL语句的灵活性，又能够满足系统在数据处理方面的实际需求。

此外，MyBatis对复杂查询、分页处理及多表关联查询均具有较好的适配能力，为本系统后续实现统计分析、状态查询和业务记录追踪等功能提供了良好的技术支持。

### 2.5 Spring Security

Spring Security是Spring生态中常用的安全框架，主要用于实现系统身份认证与访问授权控制。其核心功能包括用户身份校验、请求过滤、权限判定和安全上下文维护等，能够为Web系统提供较为完善的安全保障机制。

本系统在安全设计中引入Spring Security，对不同角色用户的访问行为进行统一控制。系统根据用户身份信息完成登录认证，并结合角色权限划分限制用户可访问的功能范围。通过安全框架的接入，系统能够有效防止未授权访问和越权操作，提升系统在用户认证和权限管理方面的规范性与可靠性。

结合本系统的实际业务需求，系统将用户划分为超级管理员、管理员、教师和学生四类角色，不同角色在设备管理、预约审批、借还处理和统计分析等功能中的操作权限存在差异。借助Spring Security，可较好地实现角色权限的分层控制，保证系统业务流程的安全有序运行。

### 2.6 前端技术

#### 2.6.1 Vue 3

Vue 3是一种渐进式JavaScript前端框架，具有组件化开发、响应式数据绑定和虚拟DOM等特点。与传统页面开发方式相比，Vue 3能够更高效地组织复杂页面结构，并提高界面交互的灵活性。其组合式API进一步增强了逻辑复用能力，使前端代码在结构上更加清晰。

本系统前端采用Vue 3进行开发，主要用于构建登录页面、设备管理页面、预约管理页面、消息中心页面以及统计分析页面等功能界面。通过Vue 3的组件化机制，系统前端能够实现页面模块复用，从而提升开发效率与后期维护便利性。

#### 2.6.2 Vite

Vite是一种面向现代前端开发的构建工具，具有启动速度快、热更新效率高、配置相对简洁等特点。相比传统构建工具，Vite在开发阶段能够更快地完成项目启动和页面更新，有助于提升前端开发效率。

在本系统中，Vite主要承担前端项目构建与本地开发服务的作用。其代理功能可以较好地支持前后端分离开发模式下的接口联调，便于系统开发与测试工作的开展。

#### 2.6.3 Element Plus

Element Plus是基于Vue 3生态的桌面端组件库，提供了表单、表格、按钮、弹窗、分页、菜单等常用组件，适用于后台管理系统界面的快速构建。该组件库风格统一、组件丰富，能够有效降低前端界面开发成本。

本系统管理端页面采用Element Plus进行界面设计，通过其现成组件快速实现设备列表展示、预约审批、表单录入、消息展示和统计分析等功能，不仅提高了界面开发效率，也保证了整体风格的一致性。

#### 2.6.4 Vue Router与Pinia

Vue Router是Vue官方提供的路由管理工具，主要用于实现单页应用中的页面跳转与路由控制。Pinia是Vue 3推荐的状态管理工具，适用于维护用户登录状态、角色信息和全局共享数据。

本系统通过Vue Router实现不同功能页面之间的导航切换，并结合权限控制机制对用户可访问页面进行限制；通过Pinia对用户身份信息和认证状态进行统一管理，从而保证系统在前端层面的状态一致性与访问安全性。

### 2.7 数据库与缓存技术

#### 2.7.1 MySQL

MySQL是一种应用广泛的开源关系型数据库管理系统，具有性能稳定、使用方便、生态成熟等特点，适用于中小型信息管理系统的数据存储需求。其支持事务机制、索引优化及多表关联查询，能够较好地满足本系统在用户信息、设备信息、预约记录、借用记录和维修记录等方面的数据管理需求。

本系统采用MySQL作为核心数据存储平台，对系统中的结构化数据进行统一管理。通过建立合理的数据表关系和约束机制，可以保证数据的一致性与完整性，为系统功能的实现提供可靠支撑。

#### 2.7.2 Redis

Redis是一种基于内存的键值型数据库，具有读写速度快、响应时间短等优点，常用于缓存、会话管理和热点数据存储等场景。在实际应用中，Redis能够有效减轻数据库访问压力，并提升系统在高频访问场景下的响应性能。

本系统在设计上预留了Redis的扩展能力，可在需要时用于缓存Token信息或系统高频访问数据，从而增强系统在并发访问情况下的性能表现。该设计也为系统后续的横向扩展提供了便利。

## 第3章 系统需求分析

### 3.1 总体需求分析

#### 3.1.1 建设背景

高校实验室设备是教学实验、课程实践和科研活动的重要基础条件，其管理水平直接影响实验教学质量和设备资源利用效率。随着高校信息化建设的不断推进，传统依赖人工登记或纸质审批的设备管理方式已难以满足现代实验室管理的实际需要。

目前，高校实验室设备管理中普遍存在以下问题：一是设备使用信息更新不及时，师生难以及时掌握设备可用状态，容易出现重复预约或资源闲置现象；二是预约、审批和借还流程较为分散，管理人员难以对相关信息进行统一汇总和全过程跟踪；三是设备借用记录不完整，发生损坏或丢失时责任追溯难度较大；四是维修申请与处理过程缺乏统一平台支撑，设备维护效率较低。

针对上述问题，设计并实现一套高校实验室设备预约管理系统具有较强的现实意义。通过信息化手段对设备信息、预约申请、审批流程、借还记录、维修处理和消息提醒等业务进行统一管理，不仅能够提高实验室设备的利用率，还能够推动实验室管理工作向规范化、数字化方向发展。

#### 3.1.2 系统总体目标

本系统面向高校实验室设备管理业务，旨在构建一套功能较为完整、流程较为规范、权限边界清晰的设备预约管理平台。其总体目标主要包括以下几个方面：

（1）构建覆盖超级管理员、管理员、教师和学生四类角色的统一管理平台，实现不同角色职责的合理划分；
（2）实现设备基础信息的统一维护、分类管理和状态跟踪，提高设备信息管理的规范性；
（3）实现预约申请、审批处理、借用登记、归还确认等业务流程的系统化管理，形成较完整的业务闭环；
（4）建立消息通知机制，对预约失效、即将到期、逾期未还及维修处理结果等关键事件进行提醒；
（5）提供基础统计分析功能，为实验室设备配置、使用监控和管理决策提供数据支持；
（6）保证系统具备一定的安全性、可维护性和可扩展性，以满足高校实验室日常管理需求。

#### 3.1.3 系统角色分析

根据系统业务特点与实验室管理实际需求，本系统设定四类用户角色，分别为超级管理员、管理员、教师和学生。不同角色在系统中承担的职责存在明显差异，其权限划分如表3-1所示。

**表3-1 系统角色及职责划分**

| 角色       | 角色代码    | 主要职责                                                       |
| ---------- | ----------- | -------------------------------------------------------------- |
| 超级管理员 | SUPER_ADMIN | 负责系统全局管理，包括管理员账号管理、角色权限配置、菜单管理等 |
| 管理员     | ADMIN       | 负责设备管理、预约终审、借用归还处理、维修管理及统计分析       |
| 教师       | TEACHER     | 负责学生预约初审、学生账号管理及个人业务操作                   |

在角色层级上，超级管理员拥有系统最高管理权限；管理员主要负责实验室设备及业务流程管理；教师主要面向所管理学生开展审核和管理；学生则作为设备预约与使用的直接参与者。该角色设计能够满足实验室管理中多层级协同工作的实际需求。

### 3.2 功能需求分析

#### 3.2.1 用户认证与账户管理需求

用户认证与账户管理是系统安全运行的基础模块，直接决定了系统能否对访问行为进行有效管控。系统面向超级管理员、管理员、教师和学生四类角色，不同角色拥有各自对应的身份标识，超级管理员和管理员通过编号或账号登录，教师通过工号登录，学生通过学号登录。系统应在用户提交登录信息后对身份标识、密码和账户状态进行逐级校验，校验通过后分配认证令牌并引导用户进入对应功能界面，校验失败时给出明确的错误提示，拒绝非法访问。

为提升账户使用安全性，系统应设计首次登录强制修改密码机制。新建账号的默认密码统一为初始值，用户首次登录成功后，系统应自动识别其首次登录状态，限制其访问业务功能页面，仅开放个人中心和密码修改入口，待用户完成密码修改并满足最低长度要求后，系统解除限制并允许正常使用全部功能。这一机制能够有效防止默认密码长期存在所带来的安全隐患。

在账户管理方面，系统应按照组织层级设计分层管理能力。超级管理员可创建、启用、禁用管理员账号并重置其密码；管理员可对其管辖的教师账号执行相同操作；教师则可创建、删除和管理其负责的学生账号。这种层级化的账户管理设计能够清晰反映高校实验室的组织结构，防止越权操作。此外，所有用户均应能够在个人中心查看自身基本信息，包括姓名、账号、角色、信用分和历史借用记录，为用户提供统一的个人信息管理入口。

#### 3.2.2 角色与权限管理需求

系统面向多类用户群体，不同角色在功能使用范围和数据访问范围方面存在明显差异，因此需要建立清晰、可配置的权限管理机制，确保每类用户的操作行为始终限定在其职责范围之内。系统应支持对角色所拥有的功能权限进行查询、分配和调整，权限粒度应覆盖主要业务操作，例如设备新增、预约审批、维修处理等，使管理员能够灵活控制各角色的功能边界。

在菜单可见范围方面，系统应根据当前登录用户的角色动态渲染侧边导航菜单，仅展示该角色有权访问的功能页面入口，隐藏无权访问的菜单项，从而降低用户因误操作进入无关页面的概率。菜单与权限应保持独立配置，菜单可见不等于自动拥有对应功能的操作权限，两者需分别进行分配和管理。

超级管理员应能够通过角色权限配置页面查看系统中所有角色的当前权限集合，并根据实际管理需求对权限项和菜单项进行调整。角色权限的变更应在下次请求时生效，以保证权限策略能够及时落地。通过构建基于角色的访问控制体系，系统能够在保证使用便利性的同时，有效防止越权访问和数据泄露风险。

#### 3.2.3 设备管理需求

设备管理是本系统的核心基础功能，其信息质量直接影响预约、借用和维修等业务模块的正常运转。系统应支持对实验室设备的基础信息进行统一维护，管理范围包括设备名称、设备编号、所属类别、当前状态、存放位置、设备图片和描述说明等字段。管理员可对设备信息进行新增、编辑和停用操作，设备编号应具备唯一性约束，防止重复录入。对于设备图片，系统应支持文件上传方式录入，并在设备详情页面提供可视化展示，增强设备信息的直观性。

设备状态管理是该模块的核心能力之一。设备在系统中的状态并非静态不变，而是随预约、借用、归还和维修等业务流程的推进而动态变化。系统应维护完整的设备状态流转机制：设备初始状态为可用，预约审批通过后转为已预约，用户取用后转为借出中，归还且完好时恢复为可用，归还时存在损坏则转为损坏状态，进入维修流程后转为维修中，维修完成后再次恢复可用。通过状态的实时联动更新，系统能够准确反映每台设备的当前实际状态，避免因信息滞后导致的设备冲突或重复预约问题。

在设备检索方面，系统应支持按设备名称关键字、设备编号关键字、类别和状态等多条件组合筛选，并以分页方式返回查询结果，避免一次性加载大量数据影响页面响应速度。管理员可通过筛选功能快速定位目标设备并执行相应操作，学生和教师则可通过设备列表查询当前可预约的设备信息，提高设备资源的利用效率。

#### 3.2.4 预约管理需求

预约管理是系统业务流程的起始环节，其设计质量直接决定了设备资源能否被合理分配和高效利用。系统应支持学生和教师用户在确认设备当前可用后提交预约申请，申请内容包括目标设备、预约开始时间、预约结束时间和使用目的。为保证同一设备不被重复占用，系统应在接受预约申请前执行时间冲突检测，对于与已有有效预约存在时间段重叠的申请，系统应明确拒绝并提示冲突原因，引导用户调整预约时间。

在审批流程设计上，系统应根据申请人的角色实施分层审批机制，以适应高校实验室管理的实际组织结构。学生提交的预约申请需先由所属教师进行初步审核，教师审核通过后方可进入管理员终审环节；而教师提交的预约申请则可直接进入管理员审批阶段，减少不必要的审批层级。审批操作应支持通过和驳回两种结果，驳回时要求填写审批意见，以便申请人了解被驳回的原因并做出调整。

终审通过后，系统应自动生成对应的借用记录，并将设备状态更新为已预约，为后续取用操作做好数据准备。预约申请在提交后、进入最终执行状态前，申请人应可主动取消，取消后设备状态恢复为可用。此外，系统应设计预约自动失效机制，对于终审通过后超过规定时间仍未被取用的预约记录，系统通过定时任务将其标记为已失效，释放设备资源，同时向申请人推送失效提醒，避免设备长期被无效占用。

#### 3.2.5 借用归还管理需求

借用归还管理是预约审批完成后的执行环节，负责记录设备从实际取用到最终归还的完整过程，是保障设备流转可追溯的重要机制。系统应在预约终审通过后自动生成借用记录，记录内容包括借用人、借用设备、关联预约、预计归还时间等基础信息，初始状态为待取用。管理员在用户实际领取设备时应在系统中执行取用确认操作，系统同步记录实际取用时间并将设备状态更新为借出中，实现线下行为与系统数据的一致性。

在设备归还环节，管理员应在用户归还设备时录入归还时间和设备状况说明。若设备归还时状态完好，系统将借用记录标记为已归还，并将设备状态恢复为可用；若归还时设备存在损坏，系统在更新借用记录状态的同时将设备标记为损坏状态，并可自动触发维修申请流程，确保损坏设备的处理不被遗漏。借用记录的设备状况字段为后续责任追溯提供了依据，有助于规范用户的借用行为。

系统还应具备逾期自动识别能力。当借用记录的预计归还时间到达后仍未办理归还手续，系统通过定时扫描任务将相应记录状态更新为逾期，并同步向借用人发送逾期提醒通知，督促其及时归还设备。对于临近归还时间但尚未逾期的借用记录，系统还应提前生成到期提醒，帮助借用人合理安排归还时间，从整体上提高设备的流转效率和归还及时率。

#### 3.2.6 消息通知需求

消息通知模块是提升系统信息传递效率的重要组成部分，通过在关键业务节点向用户推送系统消息，可以有效减少因信息滞后而导致的管理问题。系统应在以下场景自动生成并推送相应通知：用户首次登录时推送密码修改提醒；管理员重置某用户密码后推送密码重置通知；预约记录因未及时取用而自动失效时推送失效提醒；借用记录临近到期时推送到期提醒；借用记录超期后推送逾期提醒；维修申请状态发生变更时向申请人推送处理结果通知。这些通知覆盖了系统主要业务链路的关键节点，能够让用户及时感知与自身相关的业务进展。

系统应为用户提供统一的消息中心入口，集中展示所有推送给当前用户的通知信息，按时间倒序排列，最新通知优先展示。消息应分为未读和已读两种状态，用户可在消息中心对单条消息进行已读确认，系统记录确认时间。导航栏应展示当前用户未读消息的数量角标，使用户无需进入消息中心也能感知到待处理的通知信息，提高消息的触达效果。

为防止相同事件重复推送干扰用户，系统应在生成通知时判断是否已存在相同类型且关联相同业务对象的未读通知，若已存在则跳过本次推送，保证消息的不重复性。消息内容应包含清晰的标题、正文说明和关联业务类型，使用户能够快速理解通知含义并采取相应行动。

#### 3.2.7 维修管理需求

当设备在借用或日常使用过程中发生故障或损坏时，系统应提供规范化的维修申请和处理流程，确保设备损坏信息得到及时记录和妥善处置。用户在发现设备损坏后应能够在系统中提交维修申请，填写内容包括损坏设备和故障描述，系统接收申请后将设备状态更新为维修中，防止其在维修期间被其他用户预约或借用。为避免同一设备被重复提交维修申请，系统应在接受新申请前检查目标设备是否已存在处理中的维修记录，若有则拒绝重复提交并给出提示。

在维修处理环节，管理员应能够对维修记录进行全程跟踪管理。管理员可将维修状态由待处理更新为处理中，表示维修工作已经启动；维修完成后将状态更新为已完成，系统同步将设备状态恢复为可用；若设备经评估无法修复，则将维修状态标记为无法修复，设备维持损坏状态，管理员可根据实际情况决定后续处置方式，如停用或报废。每次状态变更时，管理员应填写处理说明，使申请人能够了解维修的具体进展和结论。

维修状态变更完成后，系统应向维修申请人自动推送通知，告知其维修结果，增强维修流程的透明性和用户体验。通过建立从损坏报告、维修处理到结果反馈的完整管理闭环，系统能够规范实验室设备的维护行为，减少设备损坏信息遗漏的风险，提高设备的整体维护水平和使用寿命。

#### 3.2.8 统计分析需求

统计分析模块为管理人员提供数据支撑，帮助其全面掌握系统运行状态和设备资源使用情况，从而为实验室管理决策提供参考依据。系统应提供业务数据概览功能，将设备总量、当前可用设备数、借用中设备数、预约总量、借用总量和维修总量等核心指标以数据卡片形式集中展示，使管理人员无需逐一翻阅业务列表即可快速掌握系统整体运行情况，在登录首页直接获取关键数据。

在排行榜统计方面，系统应提供热门设备榜、设备损坏榜和用户违规榜三类统计功能，并支持按累计、近半年和近一个月三种时间维度切换查看。热门设备榜通过统计各设备的借用次数，帮助管理人员识别使用频率最高的设备，为合理安排设备维护计划和采购优先级提供参考。设备损坏榜通过汇总各设备的维修申请次数，识别故障率较高的设备，辅助管理人员评估设备使用寿命和维保需求。用户违规榜通过统计各用户的逾期归还次数和损坏设备次数，帮助管理人员识别借用行为不规范的用户，作为信用管理和行为约束的数据依据。

管理人员还应能够灵活控制排行榜的展示数量，支持选择前3、前5或前10名进行展示，以适应不同场景下的数据查看需求。各统计数据应基于系统历史业务记录实时计算生成，保证统计结果的准确性和时效性。通过这一系列统计分析功能，系统能够将分散的业务数据转化为有价值的管理洞察，帮助实验室管理人员更科学地开展设备资源配置、使用监控和制度优化工作。

### 3.3 非功能需求分析

#### 3.3.1 安全性需求

系统应具备较完善的安全控制机制。首先，应通过身份认证机制保证只有合法用户能够访问系统资源；其次，应通过角色权限控制防止越权操作；再次，应对用户密码进行加密存储，降低信息泄露风险。此外，首次登录强制修改密码也是保障系统安全的重要措施之一。

#### 3.3.2 数据一致性需求

在预约审批、借用归还、维修处理等业务中，往往涉及多个数据表或多个业务状态的联动变化，因此系统应保证数据更新过程的一致性。对于关键业务操作，应采用事务控制方式，避免因部分操作失败而导致数据不一致。同时，应通过外键关联和必要约束增强数据完整性。

#### 3.3.3 可用性与性能需求

系统应具有较好的可用性，能够满足实验室设备日常管理工作的连续使用需求。对设备列表、预约记录等高频访问功能，应支持分页查询，以提高界面加载速度并减轻系统压力。在中小规模高校实验室应用场景下，系统应能够保证常规操作具有较快的响应速度。

#### 3.3.4 可扩展性需求

随着实验室管理需求的发展，系统在后续应用中可能会增加新的角色、业务流程或统计功能。因此，在系统设计阶段应充分考虑模块解耦与接口扩展能力，使系统能够在不大幅修改原有结构的前提下完成功能扩展。

#### 3.3.5 可维护性需求

系统在软件结构上应遵循清晰的分层思想，使控制层、业务层和数据访问层职责明确，以降低后续维护难度。同时，代码规范、数据库设计规范以及统一的异常处理机制也应作为系统可维护性的重要保障。

# 第4章 系统设计

## 4.1 系统架构设计

本系统采用前后端分离的B/S（Browser/Server）架构模式。用户通过浏览器访问系统前端界面，前端负责页面展示、交互处理与请求发起；后端负责业务逻辑处理、权限控制和数据持久化；数据库用于存储系统运行过程中产生的各类业务数据。系统整体架构如图4-1所示。

在该架构下，前端基于Vue 3构建单页应用，后端基于Spring Boot整合Spring MVC、Spring Security和MyBatis构建RESTful服务接口，前后端之间通过HTTP协议进行通信，数据交换格式统一采用JSON。该架构具有职责划分清晰、开发效率较高、维护与扩展较为方便等优点，适合高校实验室设备预约管理系统的建设需求。

从后端内部结构来看，系统遵循典型的分层设计思想，主要分为控制层、业务层和数据访问层。控制层负责接收客户端请求、完成参数校验并组织响应数据；业务层负责实现系统核心业务逻辑，包括预约审批、借用归还、权限校验和状态流转等；数据访问层负责与数据库进行交互，完成业务数据的增删改查操作。通过分层设计，系统各部分之间的耦合度得到有效降低，有利于后续维护与功能扩展。

此外，系统在安全控制方面引入统一认证与授权机制，在数据存储方面采用MySQL作为核心关系型数据库，并预留Redis扩展能力，以满足后续性能优化和分布式部署需求。

## 4.2 功能模块设计

根据系统需求分析结果，结合高校实验室设备管理的业务特点，系统主要划分为认证登录、用户管理、角色权限管理、设备管理、预约管理、借用归还管理、维修管理、消息通知管理、统计分析和菜单配置等功能模块。

认证登录模块主要负责用户身份认证、登录状态维护和密码修改等功能，是系统运行的基础模块。用户管理模块主要用于实现不同角色用户的创建、启用、禁用和信息维护。角色权限管理模块用于维护系统角色对应的功能权限和菜单可见范围，从而确保不同用户只能访问其职责范围内的资源。

设备管理模块主要用于维护实验室设备的基础信息，包括设备名称、编号、类别、状态、位置和描述等内容，同时支持设备信息查询、编辑和状态调整。预约管理模块是系统的核心模块之一，主要负责设备预约申请、审批流转、预约取消和预约失效处理等业务。借用归还管理模块在预约审批通过后记录设备借用情况，并负责归还登记和逾期管理。维修管理模块用于处理设备损坏后的报修、维修状态更新及维修结果维护。消息通知模块用于向用户推送预约失效、归还提醒、逾期通知和维修结果等业务消息。统计分析模块则通过对设备、预约、借用和维修等数据进行汇总分析，为管理人员提供数据支持。

各模块之间并非相互独立，而是存在密切的业务关联。例如，预约管理依赖设备管理模块提供设备状态信息，借用归还管理依赖预约审批结果生成借用记录，维修管理又与借还过程中的设备损坏情况相关联，消息通知模块则贯穿多个业务模块的关键节点。通过这种模块化设计，系统在保持结构清晰的同时，也能够较好地支撑完整的实验室设备管理业务流程。

## 4.3 界面设计

### 4.3.1 整体布局设计

为了满足后台管理系统在操作效率和信息展示方面的需求，本系统前端界面采用较为常见的管理端布局方式，即左侧导航栏、顶部操作栏和主内容区相结合的结构。左侧导航栏用于展示系统功能菜单，并根据当前登录用户的角色动态显示其可访问的菜单项；顶部操作栏主要展示系统名称、当前用户信息和退出登录等操作入口；主内容区则用于加载和显示各功能页面。

该布局方式具有结构清晰、操作路径明确和扩展性较好的特点，能够满足实验室设备管理系统在多模块切换和多角色使用场景下的界面需求。

### 4.3.2 登录页面设计

登录页面是用户进入系统的入口，界面设计以简洁明了为原则，主要包括系统名称、身份标识输入框、密码输入框和登录按钮等基本元素。不同角色用户均通过统一登录入口进入系统，后台根据用户身份信息判断其角色并分配相应权限。

对于首次登录用户，系统在登录成功后会提示修改默认密码，以提升账户安全性。该设计既保证了登录流程的统一性，也兼顾了系统安全需求。

### 4.3.3 设备管理页面设计

设备管理页面主要用于展示和维护实验室设备信息，页面由查询区域和数据展示区域两部分组成。查询区域支持通过设备名称、设备编号、类别和状态等条件进行组合筛选；数据展示区域则以表格形式展示设备的编号、名称、类别、状态和存放位置等关键信息，并在操作列中提供编辑、停用等功能入口。

在页面交互设计上，新增设备和编辑设备操作通常采用弹窗表单形式实现，便于用户在不离开当前页面的情况下完成数据维护。对于设备状态，系统通过不同样式进行区分，以增强信息辨识度。

### 4.3.4 预约管理页面设计

预约管理页面根据用户角色的不同呈现不同的功能重点。对于学生用户，页面主要包含设备预约申请入口和个人预约记录列表；对于教师用户，除个人预约记录外，还需展示待审核学生预约信息；对于管理员用户，则主要展示待终审预约记录和审批操作入口。

为便于用户了解预约详情，系统在预约记录中提供详情查看功能，可展示申请人信息、预约设备、预约时间、使用目的及审批意见等内容。该设计有利于提高审批效率，并增强业务过程的透明性。

### 4.3.5 借用归还页面设计

借用归还页面主要面向管理员开放，用于处理设备取用确认和归还登记业务。页面以借用记录列表为核心，支持按照借用状态进行筛选，如待取用、借用中、已归还和已逾期等。管理员可在对应记录中执行取用确认或归还登记操作。

在归还登记过程中，系统要求填写设备归还状况，以便根据实际情况自动更新设备状态或触发维修申请流程。该页面设计能够较好地支撑借还业务的规范化管理。

### 4.3.6 消息中心页面设计

消息中心页面用于集中展示系统向用户推送的各类通知信息，包括预约失效提醒、借用到期提醒、逾期提醒、密码重置提醒和维修处理结果通知等。页面采用列表或卡片形式展示消息内容，并按时间顺序排列，便于用户快速查看最新通知。

对于未确认消息，系统通过明显标识进行区分，并提供确认操作，以便用户及时处理相关信息。该设计有助于提高用户对系统关键事件的响应效率。

### 4.3.7 统计分析页面设计

统计分析页面主要面向管理人员，用于展示系统运行过程中的关键统计数据。页面通常由概览区域和排行榜区域组成。概览区域展示设备总数、预约总量、借用总量和维修总量等核心指标；排行榜区域则展示热门设备、设备损坏情况和用户违规情况等统计结果。

通过图表、数据卡片或表格等形式展示统计结果，可以帮助管理人员更直观地掌握系统运行情况，从而为实验室设备管理决策提供参考依据。

## 4.4 数据模型设计

### 4.4.1 概念模型设计

为保证系统业务数据能够被规范存储和高效管理，本系统采用关系型数据库进行数据建模，数据库名称为`lab_booking`。根据系统业务需求，数据库主要包括用户与权限相关表、设备与预约相关表、借用归还相关表、维修管理相关表、消息通知表以及定时任务日志表等。

在用户与权限部分，系统主要通过用户表、角色表、权限表及其关联表建立角色访问控制机制。用户表用于存储系统中各类用户的基本信息、登录信息和角色信息；角色表和权限表则用于维护系统中不同角色所具有的权限集合，从而构建较为完整的RBAC权限控制模型。

在业务核心数据部分，设备表用于存储实验室设备的基础信息和当前状态；预约记录表用于记录设备预约申请、审批意见和预约状态等信息；借用记录表用于记录设备实际借用与归还情况；维修记录表用于记录设备维修申请、处理过程和处理结果；消息通知表用于记录系统向用户发送的各类通知信息。

从表关系来看，用户表与预约表、借用表、维修表和通知表之间均存在关联关系，设备表则分别与预约表、借用表和维修表形成关联，而预约表与借用表之间构成一对一或紧密关联关系。通过建立这些数据关系，可以较好地支持设备预约、借用归还、维修处理和消息通知等完整业务流程。

在数据库设计过程中，系统通过主键、唯一约束、外键及状态字段等方式保证数据的唯一性、完整性和一致性，为系统的稳定运行提供了良好的数据基础。

## 4.5 接口设计

### 4.5.1 接口规范设计

系统采用前后端分离架构，后端需向前端提供统一规范的接口服务。本系统后端接口整体遵循RESTful设计风格，以资源为中心组织接口路径，以HTTP请求方法区分操作类型：GET用于数据查询，POST用于数据新增，PUT用于数据更新，DELETE用于数据删除。所有接口的基础路径统一为`/api`，数据传输格式统一采用JSON，时间字段统一使用`yyyy-MM-dd HH:mm:ss`格式。

**（1）认证方式**

系统采用基于令牌的无状态认证机制。用户登录成功后，后端返回令牌字符串，客户端在后续每次请求中通过HTTP请求头`Authorization: Bearer <token>`携带令牌。后端认证过滤器从请求头中提取令牌并完成身份校验，校验通过后将用户信息写入安全上下文，供后续接口处理使用。

**（2）统一响应结构**

所有接口返回统一的JSON结构，包含业务状态码`code`、提示信息`message`和业务数据`data`三个字段。其中`code`为0时表示操作成功，非0时表示业务异常。对于分页查询接口，`data`字段中包含`list`（数据列表）、`pageNum`（当前页码）、`pageSize`（每页条数）和`total`（总记录数）四个固定字段。响应结构示例如下：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],
    "pageNum": 1,
    "pageSize": 10,
    "total": 100
  }
}
```

**（3）异常响应机制**

接口层设计了统一的全局异常处理机制。对于常见异常场景，系统返回对应的HTTP状态码与业务错误码：参数校验失败返回400，未认证访问返回401，权限不足返回403，资源不存在返回404，业务状态冲突（如时间冲突、状态非法）返回409。通过统一的错误响应结构，前端能够快速定位问题原因，提升联调效率。

# 第5章 系统模块详细设计与实现

## 5.1 项目结构设计

为了提高系统代码组织的规范性和可维护性，本系统在实现过程中采用前后端分离的项目结构。后端部分按照控制层、业务层、数据访问层、实体对象层和配置层进行组织；前端部分则按照入口文件、路由模块、状态管理模块、页面视图、布局组件和公共组件进行划分。

后端项目中，控制层主要负责接口请求的接收与响应；业务层负责系统核心逻辑的处理；数据访问层负责对数据库进行操作；配置层用于统一管理系统认证、安全、数据库连接和静态资源访问等通用功能。通过这种结构设计，后端代码职责更加明确，有利于后续模块扩展和问题定位。

前端项目中，页面视图用于实现具体业务功能界面，路由模块负责页面跳转和访问控制，状态管理模块负责维护用户认证信息和全局共享数据，公共组件则用于实现不同页面之间可复用的界面元素。该结构能够较好地支撑中后台管理系统的开发需求。

## 5.2 认证模块设计与实现

认证模块是系统安全运行的基础，主要负责用户登录、身份识别、登录状态维护及密码修改等功能。系统采用基于令牌的无状态认证方式，结合Spring Security过滤链完成请求级别的身份校验，并通过首次登录强制改密机制提升账户安全性。

### 5.2.1 后端设计与实现

后端认证功能由`AuthController`、`AuthService`和`AuthTokenFilter`三个核心组件协同完成。`AuthController`对外暴露`POST /api/auth/login`、`GET /api/auth/me`和`PUT /api/auth/password`三个接口，接收请求并将处理委托给`AuthService`。

`AuthService`中的`login`方法负责完整的登录校验流程：首先通过`AuthRepository`根据`loginId`或`account`查询用户信息，若账号不存在则直接返回401错误；其次比对密码明文，密码错误同样返回401；再次检查账户的`status`字段，若为`DISABLED`则返回403拒绝访问。三项校验全部通过后，调用`AuthRepository.createToken`生成唯一令牌字符串，将令牌与用户ID的映射关系存入`InMemoryTokenStore`（或在Redis启用时切换至`RedisTokenStore`），最终将令牌、过期时间、首次登录标记及用户基本信息组装为JSON返回给客户端。

请求过滤依赖`AuthTokenFilter`，该类继承自Spring的`OncePerRequestFilter`，在每次请求到达业务控制器之前执行一次身份提取逻辑。过滤器从请求头`Authorization`字段中截取`Bearer`后的令牌字符串，通过`AuthRepository.findByToken`查找关联用户，若命中则将用户对象封装为`UsernamePasswordAuthenticationToken`并写入`SecurityContextHolder`，使后续所有业务层均可通过`authService.currentUser()`直接获取当前登录用户。

首次登录强制改密机制也在`AuthTokenFilter`中实现。当安全上下文中的用户`firstLoginRequired`字段为`true`，且请求路径不属于白名单（`/api/auth/me`和`/api/auth/password`）时，过滤器直接截断请求，返回`code:403`的业务错误响应，阻止用户访问其他功能接口。`changePassword`方法在密码修改成功后会将`firstLoginRequired`置为`false`并持久化，使该限制自动解除。

### 5.2.2 前端设计与实现

前端认证功能以`LoginView.vue`为入口界面，使用`useAuthStore`统一管理认证状态。`LoginView.vue`通过Vue 3的`reactive`管理表单数据，结合Element Plus的`ElForm`组件实现必填校验，用户点击登录后调用`authStore.signIn`方法发起登录请求。若服务端返回`firstLoginRequired=true`，页面通过`ElMessage.warning`提醒用户修改密码；登录成功后根据路由中的`redirect`参数跳转至目标页面。

认证状态由`useAuthStore`（基于Pinia实现）统一维护，其内部`state`对象包含`session`（令牌及用户简要信息）、`currentUser`（完整用户信息）和`initialized`（初始化标记）三个字段。`signIn`方法在接口调用成功后，将响应数据同步写入`localStorage`（键名`admin-auth-session`），令牌单独以键名`token`存储，实现跨页面刷新的状态持久化。`signOut`方法调用时清除全部本地存储项并重置`state`。`ensureCurrentUser`方法在路由守卫中被调用，确保每次页面切换时当前用户信息处于最新状态，若接口返回401则自动触发退出流程。

## 5.3 用户管理模块设计与实现

用户管理模块主要用于完成系统内不同角色用户的创建、查询、启用、禁用和密码重置等操作。本系统存在超级管理员、管理员、教师和学生四类角色，模块在实现过程中通过层级校验严格控制操作边界，保证管理权限不被越级使用。

### 5.3.1 后端设计与实现

用户管理后端以`UserController`为入口，将请求分发至`UserService`处理。`UserService.listUsers`方法首先调用`requireRoles`校验当前用户角色，仅允许超级管理员、管理员和教师访问；随后从内存中获取全量用户列表，通过`visibleTo`方法过滤出当前用户有权查看的下级用户，再依据`keyword`、`roleCode`和`status`条件进行二次过滤，最后进行内存分页返回。该方式避免了对数据库的每次全量查询，在中小规模数据量下具有较好的响应性能。

在用户创建方面，系统针对不同角色分别提供`createAdmin`、`createTeacher`和`createStudent`三个方法，各方法内部通过`requireRoles`强制要求调用方必须具备对应权限。新建用户的默认密码统一设为`000000`，且`firstLoginRequired`初始化为`true`，强制首次登录后修改密码。创建管理员账号后，系统同时向目标用户写入一条`FIRST_LOGIN_PASSWORD_CHANGE`类型的消息通知。

账户状态管理由`updateUserStatus`方法完成，执行前同样通过`visibleTo`校验操作目标是否属于当前用户的管辖范围，防止越权操作。密码重置由`resetPassword`方法处理，重置后将密码设为初始值，`firstLoginRequired`重新置为`true`，并通过`NotificationService`向目标用户发送`PASSWORD_RESET`类型的消息通知。删除学生账号采用逻辑删除方式，将`deleted`字段置为`true`，同时校验该学生名下是否存在未归还的借用记录，若存在则拒绝删除。

### 5.3.2 前端设计与实现

前端用户管理页面由`UserManagementView.vue`实现。页面顶部为查询区域，提供关键字、角色和状态三个筛选条件，支持组合搜索；下方为Element Plus的`ElTable`组件展示的分页用户列表，包含用户名、账号、角色、状态和信用分等字段。

页面通过`computed`属性`canManage`和`canCreate`根据当前登录用户的角色动态控制操作按钮的显示与可用状态。新增用户和编辑用户操作均通过`ElDialog`弹窗表单实现，表单项根据创建角色的不同动态调整必填字段。启用/禁用操作通过`ElMessageBox.confirm`二次确认后调用对应接口，密码重置同样通过确认弹窗保护，防止误操作。

## 5.4 设备管理模块设计与实现

设备管理模块是系统核心基础模块之一，主要用于实现实验室设备信息的维护与状态管理。系统支持设备信息的新增、查询、编辑和停用等功能，并通过设备状态与业务流程的联动更新，实时反映设备的实际可用情况。

### 5.4.1 后端设计与实现

设备管理后端由`DeviceController`和`DeviceService`构成。`DeviceService.listDevices`方法通过`DeviceMapper.selectAll`加载全量设备列表，在内存中进行关键字模糊匹配（匹配`deviceName`、`deviceCode`和`location`字段）、类别精确过滤和状态精确过滤，随后按`deviceId`升序排列并完成内存分页，以`list`、`total`、`pageNum`和`pageSize`四个字段组装响应结果。

设备新增和编辑均通过`createDevice`和`updateDevice`方法处理。两者在保存前均调用`DeviceMapper.countByDeviceCode`检查设备编号的唯一性，若与已有记录重复则抛出业务异常。新增设备时，`DeviceService`调用`DeviceMapper.selectNextDeviceId`获取自增ID，保证主键不重复。

设备图片上传由独立的`importDevice`方法（通过`DeviceImportController`暴露）处理，接收`MultipartFile`类型的图片文件，将其保存至配置项`app.storage.upload-dir`指定的本地目录下，以`deviceId`作为文件名，并将相对访问路径写入`lab_device`表的`image_url`字段。静态资源访问通过`StaticResourceConfig`将`/uploads/**`路径映射至本地目录，前端可直接通过URL访问设备图片。

设备状态联动在各业务模块中实现：`ReservationService`在终审通过后将设备状态更新为`RESERVED`；`BorrowRecordService`在确认取用时更新为`BORROWED`；归还且完好时恢复为`AVAILABLE`；归还时损坏则更新为`DAMAGED`；`RepairService`在维修完成后恢复为`AVAILABLE`。通过这种分布式状态联动，设备表始终准确反映设备当前状态。

### 5.4.2 前端设计与实现

前端设备管理页面由`DeviceManagementView.vue`实现。页面通过`computed`属性`canManage`判断当前用户是否为超级管理员或管理员，控制新增、编辑和状态调整按钮的可用性；`canDelete`属性进一步将删除权限限定为超级管理员。筛选区域使用`reactive`管理`filters`对象，监听关键字、类别、状态变化，变化时触发`loadDevices`重新请求设备列表。

设备列表通过`ElTable`展示，状态字段通过`ElTag`组件以不同颜色区分（可用为绿色、借出为橙色、停用为灰色等），增强视觉辨识度。新增和编辑操作共用一个`ElDialog`弹窗表单，通过`saveForm.deviceId`是否为空区分新增和编辑两种模式，表单项包括设备名称、编号、类别、位置、图片URL和描述等字段，提交前通过`ElForm`验证必填项。设备详情通过另一个只读弹窗展示，包含设备图片的预览。

## 5.5 预约管理模块设计与实现

预约管理模块是系统最核心的业务模块，涵盖预约申请、分级审批、主动取消和自动失效等完整流程。模块的核心设计难点在于多角色审批流转与设备状态一致性的协同维护。

### 5.5.1 后端设计与实现

预约管理后端由`ReservationController`和`ReservationService`实现。`createReservation`方法在接收申请时依次执行以下校验：通过`requireRoles`确认申请人为学生或教师；从`DeviceMapper`中查询目标设备并调用`ensureReservable`确认其状态为`AVAILABLE`；解析`startTime`和`endTime`并校验结束时间晚于开始时间；最后调用`ensureNoTimeConflict`方法遍历该设备所有有效预约记录，检测新申请时间段是否与已有记录存在重叠。全部校验通过后，根据申请人角色设置初始状态：教师申请直接设为`APPROVED`，学生申请设为`PENDING`，并持久化至`lab_reservation`表。

审批流转由`approveReservation`方法处理。方法接收`action`字段（`APPROVE`或`REJECT`）并通过`switch`分支分别执行审批逻辑。驳回分支将预约状态设为`REJECTED`并记录审核意见。通过分支根据预约当前状态判断审批阶段：若当前状态为`PENDING`（教师初审）则更新为`APPROVED`；若当前状态为`APPROVED`（管理员终审）则更新为`PICKUP_PENDING`，同时自动创建`BorrowRecordEntity`写入`lab_borrow_record`表，并将设备状态更新为`RESERVED`。教师角色在审批时还额外校验预约申请人是否属于自己管辖的学生。

预约自动失效通过`SystemTaskScheduler`实现，其中`expireReservationsTask`方法以`@Scheduled(cron = "0 0/30 * * * ?")`注解每30分钟执行一次，调用`SystemTaskService.expirePickupPendingReservations`扫描所有`PICKUP_PENDING`状态且创建时间超过24小时的预约记录，将其状态更新为`EXPIRED`，恢复设备`AVAILABLE`状态，并通过`NotificationService`向申请人发送`RESERVATION_EXPIRED`类型通知。

### 5.5.2 前端设计与实现

前端预约管理页面由`ReservationManagementView.vue`实现。页面通过`computed`属性`canCreate`、`isTeacher`和`isAdmin`识别当前用户角色，动态控制各操作按钮的显示：学生和教师可见"新建预约"按钮，教师和管理员可见"审批"按钮。

新建预约弹窗使用`ElDialog`包裹`ElForm`，包含设备选择下拉框（选项从`listDevices`接口获取）、预约起止时间（使用Element Plus的`ElDateTimePicker`组件，格式为`YYYY-MM-DD HH:mm`）和使用目的输入框。表单提交时调用`createReservation`接口，接口返回409时提示时间冲突。

预约列表通过`ElTable`展示，支持按预约状态筛选。操作列根据当前用户角色和预约状态动态渲染审批、取消等按钮。审批弹窗提供通过/驳回单选，驳回时要求填写审批意见。详情弹窗以只读方式展示完整预约信息，包括申请人、设备、时间段、使用目的和历次审批意见。

## 5.6 借用归还模块设计与实现

借用归还模块负责预约审批通过后的设备实际取用与归还管理，是预约业务在执行阶段的延伸。系统在预约终审通过时自动生成借用记录，并通过定时任务处理逾期场景，形成完整的借还闭环。

### 5.6.1 后端设计与实现

借用归还后端由`BorrowRecordController`和`BorrowRecordService`实现。`listRecords`方法支持按`status`、`userId`和`deviceId`三个维度筛选，同时通过`visibleTo`方法根据当前用户角色控制可见范围：学生只能查看自己的记录，教师可查看其管辖学生的记录，管理员和超级管理员可查看全部记录。

取用操作由`pickup`方法处理。方法校验借用记录当前状态必须为`PICKUP_PENDING`，验证通过后将`status`更新为`BORROWING`，记录`pickupTime`为当前时间，并同步将设备状态更新为`BORROWED`，最后将变更持久化至数据库。

归还操作由`returnDevice`方法处理，接收`deviceCondition`归还状况字段。方法将借用记录`status`更新为`RETURNED`，记录`returnTime`，并根据`deviceCondition`的值判断设备状态：若值为`DAMAGED`，则将设备状态更新为`DAMAGED`；其余情况均恢复为`AVAILABLE`。损坏归还时，系统还自动调用`RepairService.createRepair`创建一条初始状态为`PENDING`的维修申请记录，实现借还流程与维修流程的自动衔接。

逾期处理由`SystemTaskScheduler`中的`overdueBorrowRecordsTask`方法承担，同样以每30分钟的频率执行，调用`SystemTaskService.markOverdueBorrowRecords`扫描所有`BORROWING`状态且`expectedReturnTime`已过期的借用记录，将其`status`更新为`OVERDUE`，并通过`NotificationService`向借用人发送`BORROW_OVERDUE`类型通知。每天上午8时，`reminderSummaryTask`额外执行一次，对即将到期的借用记录生成`ABOUT_TO_EXPIRE_REMINDER`提醒通知。

### 5.6.2 前端设计与实现

前端借用记录展示分布于两个页面。`ProfileCenterView.vue`个人中心页面供学生和教师查看自己的借用记录，支持按状态筛选和分页浏览，展示取用时间、预计归还时间、实际归还时间和设备状况等关键字段。

管理员侧的借用管理功能集成于设备管理页面或独立的借还列表视图中，管理员可对状态为`PICKUP_PENDING`的记录执行取用确认操作，对状态为`BORROWING`的记录执行归还登记操作。归还登记弹窗要求管理员选择设备状况（完好/损坏），选择损坏时系统弹出二次确认提示，告知用户将同时触发维修申请。页面对逾期记录通过`ElTag`的红色标签进行视觉告警。

## 5.7 维修管理模块设计与实现

维修管理模块用于处理设备在使用过程中出现的故障和损坏问题，支持从报修申请提交到维修结果反馈的全流程管理，实现设备损坏信息的规范化记录与处置。

### 5.7.1 后端设计与实现

维修管理后端由`RepairController`和`RepairService`实现。`createRepair`方法在接收报修申请时首先通过`requireRoles`确认申请人为学生角色，随后从`DeviceMapper`查询目标设备是否存在，再调用`ensureNoActiveRepair`方法检查该设备当前是否已有`PENDING`或`PROCESSING`状态的维修记录，若存在则拒绝重复提交。校验通过后创建`RepairEntity`，初始状态设为`PENDING`，并将设备状态更新为`REPAIRING`，以防止设备在维修期间被预约。

`updateRepairStatus`方法供管理员更新维修进度。方法接收目标状态（`PROCESSING`、`COMPLETED`或`UNREPAIRABLE`）和处理说明，根据不同目标状态执行对应的设备状态联动：若更新为`COMPLETED`则将设备状态恢复为`AVAILABLE`；若更新为`UNREPAIRABLE`则将设备状态保持为`DAMAGED`，表示设备无法正常使用。维修状态变更完成后，调用`NotificationService`向申请人发送维修结果通知，通知内容包含处理说明字段，使申请人可及时了解维修结论。

`listRepairs`方法同样通过`visibleTo`进行可见范围控制：学生只能查看自己提交的维修申请，教师可查看其管辖学生提交的申请，管理员可查看全部记录，支持按`status`、`deviceId`和`applicantId`条件筛选分页。

### 5.7.2 前端设计与实现

前端维修管理页面由`RepairManagementView.vue`实现。页面顶部为状态筛选区域，支持按维修状态（待处理、处理中、已完成、无法修复）过滤记录，下方为`ElTable`展示的维修记录列表，包含设备名称、申请人、故障描述、当前状态和申请时间等字段。

管理员在列表中可对待处理或处理中的维修记录执行状态更新操作，点击操作按钮后弹出`ElDialog`，要求管理员选择新状态并填写处理说明。学生可通过列表查看自己提交的维修申请的当前处理进度，但不具备状态更新权限，相关操作按钮对学生角色隐藏。详情弹窗以只读方式展示维修记录的完整信息，包括设备信息、申请人、故障描述、处理说明及状态变更时间。

## 5.8 消息通知模块设计与实现

消息通知模块负责在关键业务节点向用户推送系统消息，贯穿认证、预约、借还和维修等多个业务模块，是提升系统信息可达性的重要支撑组件。

### 5.8.1 后端设计与实现

消息通知后端由`NotificationController`和`NotificationService`实现。通知的生成采用被动调用模式，即由各业务模块在关键节点主动调用`NotificationService.createSystemNotificationIfAbsent`方法触发。该方法首先通过`NotificationMapper.countUnread`查询是否已存在同类型、同关联业务ID的未读通知，若已存在则跳过创建，避免重复推送；若不存在则构建`NotificationEntity`并持久化至`lab_notification`表。触发场景涵盖首次登录（`FIRST_LOGIN_PASSWORD_CHANGE`）、密码重置（`PASSWORD_RESET`）、预约失效（`RESERVATION_EXPIRED`）、借用逾期（`BORROW_OVERDUE`）、即将到期提醒（`ABOUT_TO_EXPIRE_REMINDER`）和逾期提醒（`OVERDUE_REMINDER`）六种类型。

`listMyMessages`方法供用户查询个人消息列表，通过`authService.currentUser()`获取当前用户ID，结合`is_read`状态和`type`类型进行过滤筛选，支持分页返回。`summary`方法返回各类未读消息的数量汇总，前端用于在导航栏渲染消息角标。`confirmMessage`方法在用户确认消息时将`is_read`字段置为`true`，并记录`read_at`时间戳。

定时任务方面，`SystemTaskScheduler`中的`reminderSummaryTask`方法以`@Scheduled(cron = "0 0 8 * * ?")`注解在每天上午8时执行，调用`SystemTaskService.generateReminderSummary`扫描即将到期和已逾期的借用记录，为相关用户批量生成提醒通知。

### 5.8.2 前端设计与实现

前端消息中心页面由`MessageCenterView.vue`实现。页面顶部提供已读/未读状态切换标签和消息类型筛选下拉框，下方以卡片或列表形式展示消息内容，按创建时间倒序排列，最新消息置顶显示。未读消息通过样式差异（加粗标题或高亮背景）与已读消息明显区分，用户可单条或全部标记为已读。

消息角标功能通过在`AdminLayout.vue`布局组件中调用`/api/notifications/summary`接口实现，将未读消息总数展示于顶部导航栏的消息入口旁。该接口在用户登录后初始调用，并在用户访问消息中心页面确认消息后刷新计数，保证角标数量与实际未读消息数量保持一致。

## 5.9 统计分析模块设计与实现

统计分析模块主要用于对系统运行过程中产生的业务数据进行汇总和分析，为管理人员提供设备使用概况和多维度排行数据，辅助实验室管理决策。

### 5.9.1 后端设计与实现

统计分析后端由`StatisticsController`和`StatisticsService`实现。`overview`方法汇总系统当前业务规模的核心指标，通过各Mapper的`selectAll`方法加载全量数据后在内存中统计：设备总量及按状态细分的各类数量、预约总量、借用总量（含逾期数）和维修总量。为减少高频调用时的重复计算开销，`overview`方法引入了`AppCacheService`缓存机制，以`statistics:overview`为键名对结果进行短时缓存，在Redis可用时使用`RedisAppCacheService`，否则降级为`InMemoryAppCacheService`。

热门设备统计由`hotDevices`方法实现，接收`rankScope`（`TOTAL`/`HALF_YEAR`/`MONTH`）和`topN`参数。方法从`BorrowRecordMapper`加载所有借用记录，根据`rankScope`过滤时间范围，通过Java Stream的`Collectors.groupingBy`按`deviceId`聚合统计借用次数，按次数降序排列后取前`topN`条记录，并关联`DeviceMapper`补充设备名称等展示字段。设备损坏统计`deviceDamageStatistics`和用户违规统计`userViolationStatistics`采用类似的聚合分析思路，分别统计各设备的维修申请次数和各用户的逾期及损坏归还次数。

### 5.9.2 前端设计与实现

前端统计分析页面由`StatisticsView.vue`实现，同时作为登录后的默认首页。页面顶部为概览卡片区域，通过`computed`属性`summaryCards`将`overview`接口返回的数据转换为包含标题、数值和描述的卡片列表，使用Element Plus的`ElCard`组件均匀排列展示设备总量、当前可用设备数、借用中设备数、预约总量、借用总量和维修总量等指标。

页面中部为排行榜控制区域，提供`rankScope`（累计/近半年/近一个月）和`topN`（3/5/10）两个筛选选项，均以`ElSelect`下拉框实现，选项变化时自动触发`loadRankings`方法重新拉取三类排行榜数据。排行榜区域以并排的`ElTable`分别展示热门设备榜、设备损坏榜和用户违规榜，支持按数值列排序，帮助管理员快速识别高频使用设备和高风险用户。

## 5.10 前端路由与权限控制实现

为保证不同角色用户只能访问其权限范围内的功能页面，前端采用路由守卫与角色元数据相结合的方式实现访问控制，并通过Pinia状态存储保障登录状态在页面刷新后的持久性。

路由配置集中于`router/index.ts`和`router/access.ts`两个文件。`access.ts`通过`AppRouteMeta`接口为每条路由定义元数据，其中`roles`字段声明允许访问该路由的角色列表，`allowFirstLogin`字段标记首次登录用户是否可访问该路由。`appRouteChildren`数组集中定义所有业务路由，各路由通过`meta`字段关联权限元数据。

路由守卫在`router/index.ts`的`router.beforeEach`钩子中实现，每次路由跳转前执行以下判断：若目标路由`meta.public`为`true`（登录页）且当前用户已登录，则直接重定向至首页；若目标路由要求认证但当前无令牌，则重定向至登录页并携带`redirect`参数；认证通过后调用`authStore.ensureCurrentUser`获取最新用户信息，再通过`canAccessRoute`方法判断当前用户角色是否满足目标路由的`roles`约束，权限不足时跳转至`/403`无权限页面；对于首次登录用户，若目标路由`allowFirstLogin`不为`true`，则同样跳转至`/403`，引导用户完成密码修改。

`getAccessibleMenuItems`函数根据当前用户角色从`appRouteChildren`中筛选出具有`menu:true`标记且角色满足条件的路由，生成侧边栏导航菜单列表，由`AdminLayout.vue`布局组件调用后动态渲染，实现菜单项随角色自动变化的效果。

# 第6章 系统测试

系统测试的核心目标在于全面验证系统功能的正确性、业务流程的完整性以及权限控制的有效性。本系统采用分层测试策略，测试范围覆盖认证登录、用户管理、设备管理、预约管理、借用归还、维修管理、消息通知和统计分析等全部业务模块。后端测试重点验证业务逻辑、接口处理和状态流转的正确性；前端测试重点验证页面渲染、路由守卫和状态管理逻辑的可靠性；功能验收测试则从实际使用场景出发，对核心业务链路进行综合验证；性能与安全测试从系统稳定性和防护能力两个维度进行评估。通过多层次测试，可以较全面地判断系统是否达到预期设计目标。

## 6.1 测试环境

本系统测试环境如下：后端运行于JDK 17环境，Web框架为Spring Boot 3，持久化框架为MyBatis，数据库为MySQL 8；前端基于Node.js 20环境，构建工具为Vite，UI框架为Vue 3与Element Plus。后端测试采用JUnit 5与Spring Boot Test框架，通过`@SpringBootTest`注解启动完整应用上下文，对各Service和Controller进行集成测试；前端测试采用Vitest工具，结合`@vue/test-utils`对视图组件和路由守卫逻辑进行单元测试。整体测试环境能够满足系统功能验证与基础性能评估的实际需求。

## 6.2 后端测试

### 6.2.1 测试内容

后端测试围绕系统全部业务模块展开，测试内容主要分为正常流程验证、异常输入校验、权限边界验证和业务状态流转验证四类。正常流程验证确保各接口在合法输入和合规操作下能够返回预期结果；异常输入校验验证系统对非法参数、空值和越界值的处理能力；权限边界验证测试不同角色对各接口的访问是否受到正确限制；状态流转验证重点关注设备状态、预约状态和借用状态在业务操作前后的一致性。

### 6.2.2 典型测试用例分析

**认证模块测试**：向`POST /api/auth/login`提交正确的学号和密码，系统返回包含令牌、`firstLoginRequired`标记和用户基本信息的响应，认证成功。随后使用错误密码重新请求，系统返回401状态码和”账号或密码错误”提示。对已被禁用的账号发起登录请求，系统返回403状态码和”账号已禁用”提示，说明账户状态校验逻辑正确。在携带有效令牌的请求头中访问`GET /api/auth/me`接口，系统返回当前用户完整信息；移除令牌后再次请求，系统返回401错误，说明认证过滤机制正常工作。

**首次登录改密限制测试**：使用新建账号（`firstLoginRequired=true`）登录成功后，携带返回的令牌访问`GET /api/devices`接口，系统返回`code:403`及”首次登录请先修改密码”提示，而非正常的设备列表，说明`AuthTokenFilter`中的首次登录拦截逻辑有效。在访问白名单接口`GET /api/auth/me`和`PUT /api/auth/password`时，系统正常响应，不受拦截，符合设计预期。

**预约时间冲突测试**：系统中已存在某设备在2026年5月10日09:00至11:00时间段内的有效预约记录，随后再次提交该设备2026年5月10日10:00至12:00时间段的预约申请，时间段存在重叠。测试结果表明，系统识别出时间冲突并返回409状态码拒绝提交，说明时间冲突检测逻辑正确。将申请时间调整为当日13:00至15:00后重新提交，系统正常创建预约记录，验证了冲突检测不影响合法预约的正常提交。

**分级审批流转测试**：学生提交预约申请后，记录初始状态为`PENDING`。以教师角色调用审批接口执行通过操作，预约状态更新为`APPROVED`，数据库中同步记录教师的审批意见和审批人ID。以管理员角色再次调用审批接口执行终审通过操作，预约状态更新为`PICKUP_PENDING`，同时系统自动在`lab_borrow_record`表中创建对应借用记录，并将设备状态更新为`RESERVED`。随后测试跳过教师初审直接由管理员终审的场景，管理员对状态为`PENDING`的学生预约执行通过操作，系统返回409状态码并提示”当前预约状态不可审核”，说明审批流程的状态前置校验逻辑正确。

**损坏归还触发维修流程测试**：管理员对状态为`BORROWING`的借用记录执行归还登记操作，`deviceCondition`字段填写”DAMAGED”。测试结果表明，系统将借用记录状态更新为`RETURNED`，设备状态更新为`DAMAGED`，同时在`lab_repair`表中自动创建初始状态为`PENDING`的维修记录，归还流程与维修流程之间的业务联动实现正确。对同一设备再次提交维修申请时，系统检测到已存在待处理的维修记录，返回业务错误拒绝重复提交，防重复校验有效。

**权限越界测试**：使用学生角色令牌访问`POST /api/devices`设备新增接口，系统返回403错误，拒绝无权限操作。使用教师角色令牌尝试审批不属于其管辖范围学生的预约申请，系统同样返回403错误，说明跨层级越权操作被有效拦截。

### 6.2.3 测试结果分析

从后端测试结果来看，系统在各主要业务场景下均能够按照预期执行。认证模块在正常登录、密码错误、账户禁用和首次登录限制等场景下的响应均符合设计要求；预约模块的时间冲突检测、分级审批流转和状态前置校验逻辑正确；借用归还模块与维修模块之间的业务联动准确无误；权限控制在角色越界和层级越权两类场景下均能有效拦截。测试结果表明，系统后端在逻辑正确性和业务完整性方面基本满足设计要求。

## 6.3 前端测试

### 6.3.1 测试内容

前端测试主要包括路由守卫测试、状态管理测试、页面渲染测试和核心业务页面交互验证四个方面。路由守卫测试验证系统在未登录、权限不足和首次登录未改密三种场景下的页面跳转行为；状态管理测试验证用户认证信息在`localStorage`中的持久化写入、读取和清除是否正确；页面渲染测试验证各功能页面在正常数据条件下能否完整渲染核心界面元素；交互验证则对设备查询筛选、预约申请提交、审批操作等关键交互流程进行功能性验证。

### 6.3.2 典型测试用例分析

**未登录访问拦截测试**：在测试环境中清除`localStorage`中的认证数据，直接通过URL访问设备管理页面`/devices`。路由守卫检测到当前无有效令牌，将页面重定向至`/login`，并在URL中附加`redirect=/devices`参数，说明登录拦截机制有效。登录成功后，系统自动跳转回原目标页面，无需用户手动导航，用户体验良好。

**角色权限拦截测试**：以学生账号登录后直接访问`/roles`角色权限配置页面。路由守卫读取当前用户角色为`STUDENT`，与该路由`meta.roles`中声明的`[“SUPER_ADMIN”]`不匹配，将页面跳转至`/403`无权限提示页面，说明前端角色权限控制逻辑正确。侧边导航菜单中同样不显示角色权限入口，`getAccessibleMenuItems`函数对菜单项的动态过滤结果符合预期。

**首次登录状态拦截测试**：使用`firstLoginRequired=true`的账号登录后，路由守卫检测到首次登录状态，访问设备管理页面时被重定向至`/403`。直接访问个人中心页面`/profile`（`allowFirstLogin:true`），系统正常展示页面，密码修改功能可用。修改密码成功后，再次访问设备管理页面，路由守卫不再拦截，页面正常展示，说明首次登录限制在密码修改后自动解除。

**认证状态持久化测试**：登录成功后刷新页面，检查`localStorage`中`admin-auth-session`键的内容，确认会话数据被正确写入。刷新后路由守卫调用`authStore.ensureCurrentUser`重新获取用户信息成功，页面正常展示，说明登录状态在页面刷新后得到正确恢复，持久化机制有效。执行退出登录操作后，`localStorage`中的认证数据被清除，再次访问受保护页面时跳转至登录页，说明退出流程清理逻辑完整。

### 6.3.3 测试结果分析

前端测试结果表明，系统路由守卫在未登录、权限不足和首次登录限制三种场景下均能正确执行跳转逻辑。认证状态的`localStorage`持久化读写及退出时的数据清除均工作正常，页面刷新后登录状态能够正常恢复。导航菜单根据用户角色动态渲染的结果准确，不同角色登录后可见的菜单项与权限配置一致。各功能页面在正常数据条件下能够稳定渲染，Element Plus组件表单验证和接口调用交互均表现正常。

## 6.5 性能与安全测试

### 6.5.1 性能测试分析

在本地测试环境下，对系统常用接口进行了基础响应时间测量。测试数据量设置为设备约50条、预约记录约200条、借用记录约100条、用户约30名，与典型高校实验室中小规模使用场景相近。在该数量级下，设备列表查询接口的响应时间约为30至60毫秒，用户登录接口约为20至40毫秒，预约记录查询接口约为40至80毫秒，统计概览接口（无缓存情况下）约为80至120毫秒。各核心接口均能在100毫秒内完成响应，满足日常使用场景下的基本性能要求。

需要说明的是，系统后端当前采用内存过滤方式实现分页查询，数据量较小时性能表现良好，但随着数据量增大，内存排序和过滤的计算代价将有所上升。本系统主要面向高校实验室中小规模应用场景，在当前数据规模下性能表现可满足实际需求。若未来业务规模扩大，可通过引入数据库分页查询替代内存分页的方式进行优化。

### 6.5.2 安全测试分析

安全测试覆盖了以下典型场景：直接访问受保护接口但不携带令牌，系统返回401状态码，拒绝未认证访问；携带格式错误或已过期的令牌访问接口，系统无法在令牌存储中找到对应用户，同样返回401错误；以学生角色令牌访问管理员专属接口（如设备新增、预约审批），系统返回403错误，拒绝低权限访问；以教师角色尝试查看或操作不属于其管辖学生的数据，业务层的`visibleTo`校验生效，返回403错误，越权访问被有效拦截。

密码安全方面，系统对新建账号和密码重置均设置了强制首次修改密码机制，`AuthTokenFilter`确保持有默认密码令牌的用户在修改密码前无法访问任何业务接口。新密码长度不足8位时，`changePassword`方法抛出业务异常并返回明确提示，防止用户设置过短的弱密码。整体来看，系统在身份认证和权限控制方面具备较好的安全保障能力。

通过后端集成测试、前端单元测试、功能验收测试以及基础性能与安全测试，本系统在主要业务功能、权限控制、状态流转和异常处理等方面均达到了预期设计目标。系统较好地实现了实验室设备预约管理的核心业务闭环，从预约申请、分级审批、借用确认到归还登记和维修处理，各环节数据流转准确，状态一致性良好。多角色权限控制在角色访问限制和数据可见范围两个维度上均表现稳定，认证机制在正常和异常场景下的响应符合设计规范。在中小规模数据量条件下，系统响应性能满足实际使用需求，基础安全防护能力可覆盖高校实验室信息管理系统的常见安全场景。综合测试结果表明，本系统已基本实现预期功能目标，具备一定的实用价值。

# 第7章 总结与展望

## 7.1 工作总结

本文研究并实现了一套基于SSM框架的高校实验室设备预约管理系统，采用Spring Boot整合Spring MVC、Spring Security和MyBatis构建后端服务，以Vue 3配合Element Plus、Pinia和Vue Router构建前端管理界面，前后端通过RESTful JSON接口进行数据交互，整体架构清晰、职责分明。

在功能实现方面，系统覆盖了高校实验室设备管理的核心业务需求。认证模块基于令牌机制实现了无状态身份认证，并设计了首次登录强制改密机制保障账户安全。用户管理模块按照超级管理员、管理员、教师和学生四类角色建立了分层创建与管理体系，各角色的操作边界通过权限校验严格约束。设备管理模块维护了设备从录入到状态联动的完整生命周期，支持图片上传和多条件检索。预约管理模块实现了基于角色的分级审批流转，并通过时间冲突检测防止资源重复占用，定时任务负责自动处理超期未取用的失效预约。借用归还模块在预约审批通过后自动生成借用记录，并通过定时扫描任务实现逾期状态的自动标记和提醒推送。维修管理模块建立了从损坏报告到修复反馈的完整管理链条，归还损坏时可自动触发维修申请。消息通知模块采用幂等创建机制，在各业务关键节点向相关用户推送通知，覆盖六类通知场景。统计分析模块提供了业务概览卡片和支持三种时间维度的热门设备、设备损坏及用户违规排行榜，为管理人员决策提供数据参考。

通过系统测试验证，各功能模块均能够按照设计预期正常运行，业务流程完整性、权限控制有效性和状态流转一致性均达到预期标准。系统在中小规模高校实验室应用场景下具备较好的实用性，能够有效替代传统人工登记和纸质审批方式，提升实验室设备管理的规范化和信息化水平。

## 7.2 系统创新点

本系统在设计与实现过程中，结合高校实验室管理的实际业务特点，形成了以下几个值得关注的设计亮点。

第一，设计了基于角色差异的自适应审批流转机制。系统并非对所有预约申请采用统一的固定审批链路，而是根据申请人角色动态决定审批起点：学生申请从教师初审开始，教师申请直接进入管理员终审阶段。这一设计在保留多层监管能力的同时，有效减少了不必要的审批层级，使审批流程更贴合实际组织结构。

第二，实现了设备状态的全业务周期联动管理。系统将设备状态（可用、已预约、借出中、维修中、损坏、停用）与预约、借用、归还和维修等各业务操作深度绑定，任一业务节点的状态变化均会实时同步至设备表，确保系统中的设备状态始终准确反映设备的真实可用情况，从根本上解决了信息滞后导致的重复占用问题。

第三，引入了幂等消息推送机制。消息通知模块在创建通知前首先检查是否已存在相同类型且关联相同业务对象的未读通知，避免因定时任务重复执行或业务并发操作导致相同消息被重复推送，在保证消息送达的同时有效控制了通知数量，提升了用户的消息体验。

第四，采用了前端路由元数据驱动的权限控制模式。通过在路由配置的`meta`字段中声明`roles`和`allowFirstLogin`等权限属性，将权限规则与路由定义集中管理，路由守卫和菜单渲染均基于相同的元数据源进行判断，消除了权限规则分散维护带来的不一致风险，使前端访问控制逻辑清晰且易于扩展。

## 7.3 不足与展望

尽管本系统已基本实现高校实验室设备预约管理的核心功能，但受开发周期和应用场景规模的限制，系统在部分方面仍存在一定不足，有待在后续工作中进一步完善。

在技术实现层面，系统当前采用基于内存的令牌存储方案，在单实例部署下运行稳定，但在多节点水平扩展场景下会出现令牌共享问题，无法直接支持分布式部署。此外，设备列表、预约记录等查询当前采用全量加载后内存过滤的方式，随数据量增长性能下降明显，需要引入数据库层面的条件查询和分页支持加以优化。统计分析模块中的概览数据虽已引入缓存机制，但三类排行榜数据每次请求均实时计算，在数据量较大时存在一定性能压力。

在功能完善层面，系统目前不支持通过Excel批量导入设备信息，设备录入效率在数量较多时有限；前端仅提供Web管理端，尚未开发移动端应用，限制了学生和教师在移动场景下的使用便利性；消息通知当前依赖用户主动进入消息中心查看，缺少浏览器推送或邮件等主动触达渠道，对于重要提醒的及时性有一定影响。

针对上述不足，对系统未来的改进方向做出以下展望。一是将令牌存储迁移至Redis集群，解决分布式部署下的会话共享问题，同时利用Redis过期机制自动处理令牌失效，降低存储维护成本。二是将关键查询接口改造为数据库层分页查询，结合适当的索引优化，提升系统在中大规模数据量下的查询性能。三是增加Excel批量导入设备的功能，支持管理员通过标准模板一次性录入大批量设备信息，降低设备初始化的操作成本。四是开发基于微信小程序或移动Web的轻量端应用，使学生和教师能够在移动设备上方便地发起预约申请、查看借用状态和接收消息通知，进一步提升系统的使用便利性。五是集成浏览器Web Push或电子邮件通知渠道，对借用到期、设备逾期等高优先级事件提供主动触达能力，提高关键消息的送达率和处理及时性。

随着高校信息化建设的持续推进，实验室设备管理系统在功能深度和技术架构上均面临更高的要求。本系统的研究与实现为高校实验室信息化管理提供了一套可参考的实践方案，后续将在上述方向持续优化，不断提升系统的稳定性、易用性和可扩展性，以更好地服务于高校实验室的日常管理工作。

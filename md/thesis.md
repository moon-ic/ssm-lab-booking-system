# 基于SSM框架的高校实验室设备预约管理系统设计与实现

## 摘要

随着信息技术的快速发展与高等教育信息化进程的持续推进，高校实验室管理工作面临设备状态信息更新不及时、预约审批流程分散、借还记录追溯困难等现实挑战。传统依赖人工登记和纸质审批的管理模式，已难以满足现代高校实验室高效、规范运营的需要。

本文设计并实现了一套基于SSM（Spring + SpringMVC + MyBatis）框架的高校实验室设备预约管理系统。系统采用前后端分离架构，后端以Spring Boot为工程化基础，结合Spring Security实现多角色权限控制，以MyBatis完成数据持久化；前端基于Vue 3与Element Plus组件库构建响应式管理界面，以MySQL作为核心数据存储。系统设计超级管理员、管理员、教师和学生四类角色，覆盖用户认证、设备信息管理、预约申请与审批、借用归还处理、维修管理、消息通知及统计分析等核心功能模块，实现了设备从预约到归还的全生命周期闭环管理。经系统测试，各功能模块运行稳定，权限控制机制有效，能够满足高校实验室设备规范化管理的实际需求，对推动高校实验室管理数字化转型具有一定的参考意义。

**关键词：** 高校实验室；设备预约管理；SSM框架；前后端分离；Spring Boot；Vue 3

## Abstract

With the rapid advancement of information technology and the continuous deepening of digital transformation in higher education, the modernization of laboratory equipment management has become an increasingly critical factor affecting the quality of teaching and scientific research. However, a considerable number of universities still rely on manual registration and paper-based approval processes, commonly facing problems such as delayed equipment status updates, fragmented reservation and borrowing workflows, difficulty in data traceability, and insufficient support for management decision-making.

To address these challenges, this paper designs and implements a university laboratory equipment reservation management system based on the SSM (Spring + SpringMVC + MyBatis) framework. The system adopts a front-end and back-end separated architecture. The back end is built on Spring Boot, integrating Spring Security to provide Role-Based Access Control (RBAC) for multi-role permission management, and uses MyBatis for data persistence. The front end is developed with Vue 3 and the Element Plus component library to build a responsive management interface, with Vue Router handling permission-based routing and Pinia managing global authentication state. MySQL serves as the primary database, with Redis reserved for cache extension. The system defines four user roles — super administrator, administrator, teacher, and student — and covers seven core functional modules: user authentication and permission management, equipment information management, reservation application and multi-level approval, borrowing and return processing, equipment repair management, message notification, and statistical analysis dashboards. The device status is dynamically linked to every business operation, forming a complete closed-loop lifecycle management from reservation to return.

Functional testing confirms that each module operates correctly according to design expectations. Cross-role permission control, business process integrity, and device status consistency all meet the anticipated standards. The system demonstrates practical utility for university laboratory management scenarios of small to medium scale, effectively replacing traditional manual and paper-based workflows. This work provides a replicable engineering reference for the development of similar higher education management information systems using a front-end/back-end separation architecture.

**Keywords:** University laboratory; Equipment reservation management; SSM framework; Front-end and back-end separation; Spring Boot; Vue 3; RBAC access control

## 第1章 引言

### 1.1 研究背景和意义

#### 1.1.1 研究背景

高校实验室是开展实践教学与科学研究的核心场所，其仪器设备的高效管理对保障实验教学质量、支撑科研活动顺利开展具有重要意义。近年来，随着国家"双一流"战略的深入推进，高校持续加大实验室建设投入，实验设备种类和数量大幅增加，实验室管理工作面临的压力也随之增大。然而，我国实验室信息管理系统的应用比例仍相对偏低，相当数量的高校主要依靠纸质表格或电子表格开展设备管理，信息化建设水平整体滞后于管理实际需要。

当前高校实验室设备管理普遍存在以下突出问题：一是设备使用信息更新不及时，师生往往难以实时掌握设备可用状态，容易出现重复预约或资源闲置现象；二是预约审批与借还流程较为分散，管理人员难以对业务信息进行统一汇总与全过程跟踪，工作效率低下；三是设备借用记录不完整，一旦发生损坏或丢失，责任追溯难度较大；四是维修申请与处理过程缺乏统一管理平台，设备维护效率较低；五是缺乏有效的数据统计与决策支持手段，管理层难以对设备利用率、损坏率等关键指标进行科学评估。

在国家积极推进教育数字化转型的背景下，构建一套信息化、规范化的高校实验室设备预约管理系统，已成为提升实验室管理水平的迫切需求。通过系统化手段整合设备信息维护、预约审批、借还管理、维修处理和统计分析等业务，不仅能够有效提升设备资源利用效率，还能推动实验室管理向规范化、数字化方向发展，为高校"双一流"建设提供有力支撑[1]。

#### 1.1.2 研究意义

本课题具有重要的实践意义与技术参考价值。

在实践层面，系统能够将传统人工管理模式转变为信息化管理模式，显著降低管理员的重复性工作量，减少因人工操作失误引发的数据错漏问题。通过实时查询和在线预约机制，师生可以便捷获取设备使用状态，合理安排借用计划，有效提高设备利用率。数字化借还记录与责任绑定机制有助于降低设备丢失风险，增强资产管理的透明度与可追溯性。统计分析功能的引入则为管理人员提供了数据支撑，有助于优化设备采购与维护决策，促进资源的科学配置，推动高校智慧校园建设[2]。

在技术层面，本系统将主流企业级后端框架（Spring Boot + MyBatis）与现代前端技术（Vue 3 + Element Plus）进行深度整合，探索了前后端分离架构在高校管理信息系统中的工程实践路径，为同类系统的设计与开发提供了可借鉴的技术参考。

### 1.2 国内外研究现状

#### 1.2.1 国外研究现状

国外高校及研究机构在实验室信息化管理领域起步较早，相关研究已从基础的数字化预约管理逐步演进至智能化、物联化方向。

在物联网智能实验室系统研究方面，Aly等人于2024年在《Journal of Advances in Information Technology》发表研究，提出了一种基于物联网（IoT）技术的高校化学实验室智能管理系统，通过部署传感器网络对实验室温湿度、有害气体浓度等环境参数进行实时监测，并将环境监测数据与设备预约调度系统相结合，在提升实验室安全管理水平的同时改善了设备资源调度效率[3]。

在人工智能与物联网融合应用方面，Zhu等人于2024年在Springer出版的国际会议论文集中发表成果，设计并实现了一套基于AIoT（人工智能物联网）技术的智能实验室管理系统，综合利用IoT设备采集实验室环境、用电和视频等多维数据，结合人工智能算法实现实验室资源的动态调度与异常预警，为实验室精细化管理提供了新的技术路径[4]。

在生物实验室智能监管方面，Liu等人于2025年在《Scientific Reports》上发表研究，提出了一套面向生物实验室的全流程智能监督系统，涵盖人员行为识别、安全网络传输和智能决策辅助等模块，实现了实验室全要素、全流程的智能化管控，为实验室信息化管理体系建设提供了新的研究视角[5]。

在商业化实验室管理平台方面，国际机构亦呈现出明显的集成化发展趋势。以Thermo Fisher iLab平台为代表的商业系统，已能够实现仪器设备预约、采购工作流与数据分析的统一管理，体现了国际实验室管理系统向平台化、一体化方向演进的整体趋势[6]。

综合来看，国外实验室信息化管理研究已积累了丰富成果，技术方向日趋向物联网感知、云计算处理和人工智能分析等前沿领域延伸，系统功能更加全面、智能。但相关系统的建设成本普遍较高，且技术门槛较大，难以直接移植至国内普通高校的实际应用场景。

#### 1.2.2 国内研究现状

近年来，国内高校实验室信息化建设步伐明显加快，相关研究成果持续涌现，研究方向主要集中在系统设计与实现、技术选型优化和智慧实验室建设等方面。

在系统设计与开发层面，杨萍萍等人于2023年在《实验科学与技术》期刊发表《基于低代码的高校实验室预约系统设计与实现》，针对传统实验室预约管理系统开发周期长、维护成本高的问题，探索了利用低代码开发平台快速构建实验室预约系统的可行性，有效降低了系统的开发门槛，为高校同类系统的快速部署提供了新思路[7]。

在大数据技术应用方面，国内学者围绕实验室设备预约管理系统的数据处理能力展开了深入研究，通过引入大数据处理技术对设备使用行为数据进行聚合分析，为实验室资源的精细化管理提供了数据驱动的决策支撑，提升了系统的统计分析价值[8]。

在智慧实验室建设方向，秦艳姣等人结合教育数字化背景，探讨了高校智慧实验室的建设路径与实践经验，指出未来实验室管理系统应重点强化移动端访问、人工智能辅助决策和物联网设备接入等能力，推动实验室管理迈向更高层次的智能化水平[2]。

从整体来看，国内现有研究在系统功能完整性和技术实现路径方面已取得显著进展，Spring Boot + Vue 前后端分离架构已成为新建高校管理信息系统的主流技术选型。但部分系统在多角色权限精细化设计、跨业务流程的闭环管理和数据统计分析能力建设等方面仍存在一定的提升空间，亟需进一步完善。本文在充分借鉴上述研究成果的基础上，结合高校实验室设备管理的实际需求，设计并实现了一套功能较为完整的预约管理系统，以弥补现有研究的不足。

## 第2章 相关技术介绍

本章介绍系统开发所采用的主要技术框架与工具，包括后端SSM技术栈（Spring + SpringMVC + MyBatis）及Spring Boot工程化实践、安全认证框架Spring Security、前端Vue 3技术体系，以及数据库与缓存方案选型。

### 2.1 SSM框架概述

SSM框架是Java Web开发领域应用广泛的经典技术组合，通常由Spring、Spring MVC和MyBatis三部分组成。Spring主要负责对象生命周期管理与业务层解耦，Spring MVC负责Web层请求分发与响应处理，MyBatis负责关系型数据库访问与持久化操作。三者协同配合，能够构建层次清晰、职责明确的企业级Web应用架构[15]。

本系统在整体架构设计上采用SSM核心思想，并以Spring Boot作为工程化实现基础。相较于传统SSM项目中较为繁琐的XML配置方式，Spring Boot通过约定优于配置（Convention over Configuration）和自动装配机制显著简化了项目初始化与运行部署流程，同时保留了Spring生态体系下的典型分层架构模式。系统后端划分为控制层（Controller）、业务层（Service）和数据访问层（Repository/Mapper）三层结构：控制层负责接收与响应客户端请求；业务层负责系统核心业务逻辑处理；数据访问层负责与数据库进行交互操作。通过这种分层设计，各功能模块之间的耦合度得以有效降低，系统的可维护性与可扩展性得到显著提升。

### 2.2 Spring框架

Spring是SSM体系中的核心框架，其主要优势体现在控制反转（Inversion of Control，IoC）和依赖注入（Dependency Injection，DI）两大核心机制上。传统开发模式下，对象的创建与依赖关系通常由开发者在代码中直接管理，容易造成模块间高度耦合；而Spring通过IoC容器统一接管对象生命周期和依赖关系的维护，将组件的创建与组装工作从业务代码中剥离出来，从而增强了系统的灵活性与可维护性。

除IoC与DI外，Spring还提供了面向切面编程（Aspect Oriented Programming，AOP）能力，能够将事务控制、日志记录、权限校验等横切关注点从业务逻辑中分离出来，以声明式方式织入目标方法执行流程。该机制有助于减少重复代码，提升系统整体开发效率与代码整洁度。

在本系统中，Spring主要承担以下职责：统一管理控制层、业务层和持久层相关Bean组件；协调各层之间的依赖关系，保证IoC容器中Bean的正常注入；为数据库事务控制和跨切面逻辑（如操作日志记录）提供基础支撑，保障系统业务操作的原子性与可靠性。

### 2.3 Spring MVC

Spring MVC是Spring生态中用于构建Web应用的核心模块，其设计遵循模型—视图—控制器（Model-View-Controller，MVC）模式，通过统一的前端控制器（DispatcherServlet）对客户端请求进行集中接收，并根据URL映射关系将请求分发至对应的控制器方法（Handler Method）进行处理，最终将处理结果封装并返回给客户端[17]。

在本系统中，由于采用前后端完全分离的B/S架构，Spring MVC主要承担接口层的组织与管理职责。前端Vue 3应用通过HTTP请求访问后端功能接口，后端控制器接收请求参数后调用业务层完成相应处理，最终将结果以JSON格式序列化返回。Spring MVC提供的参数绑定（`@RequestBody`、`@RequestParam`）、数据校验（`@Valid`）、异常统一处理（`@RestControllerAdvice`）等机制，使系统接口层代码结构规范、可读性强，有效提升了后端接口的开发效率与维护便利性。

### 2.4 MyBatis框架

MyBatis是一种轻量级Java持久层框架，其核心能力在于将SQL语句与Java方法进行灵活映射，实现关系型数据库与Java对象之间的双向转换。与Hibernate等全自动ORM框架相比，MyBatis保留了开发者对SQL语句的完整控制权，对查询逻辑和执行计划具有更强的可操控性，在业务规则明确、对SQL可读性和查询性能要求较高的场景下具有明显优势[16]。

本系统以MyBatis作为数据持久化实现框架，通过Mapper接口与XML映射文件相结合的方式完成各业务模块的数据库访问操作。对于用户、设备、预约、借用、维修和通知等核心业务数据，系统均经由MyBatis完成增删改查操作，并借助ResultMap机制实现数据库字段与实体类属性之间的自动对应转换，有效减少了手工映射的工作量。此外，MyBatis对多表关联查询、动态SQL拼接、分页处理等复杂查询场景均具备良好的支持能力，为本系统在统计分析、预约冲突检测和业务记录追踪等功能的实现提供了可靠的数据访问基础。

### 2.5 Spring Security

Spring Security是Spring生态中用于构建应用安全体系的主流框架，主要提供身份认证（Authentication）与访问授权（Authorization）两大核心能力，同时具备请求过滤、权限判定、安全上下文维护和常见攻击防护等功能，能够为Web系统提供相对完善的安全保障机制。

在权限模型设计方面，本系统采用基于角色的访问控制（Role-Based Access Control，RBAC）模型，将系统权限划分到角色粒度，再将角色分配给具体用户，从而实现权限管理的集中化和可维护性[11]。在此基础上，借助Spring Security的`OncePerRequestFilter`机制对每次HTTP请求执行令牌解析与身份提取，结合`SecurityContextHolder`维护当前请求的安全上下文，实现了无状态的令牌认证方案。此外，参考文献[16]中关于JWT与RBAC结合增强Spring Security安全性的研究思路，本系统通过自定义认证过滤器与业务层权限校验双重机制进一步加固了系统访问控制的可靠性。

结合实际业务需求，系统将用户划分为超级管理员（SUPER_ADMIN）、管理员（ADMIN）、教师（TEACHER）和学生（STUDENT）四类角色，不同角色在设备管理、预约审批、借还处理和统计分析等业务功能中的操作权限明确区分，借助Spring Security实现角色权限的分层精细化控制，保障系统业务流程的安全有序运行。

### 2.6 前端技术

#### 2.6.1 Vue 3

Vue 3是由尤雨溪团队开发的渐进式JavaScript前端框架，具有组件化开发、响应式数据绑定和虚拟DOM等特点。与传统多页面开发模式相比，Vue 3能够更高效地组织复杂页面逻辑，并通过声明式渲染机制提高界面交互的开发效率。其组合式API（Composition API）进一步增强了业务逻辑的复用能力与代码的组织灵活性，尤其适合构建中后台管理系统等功能密度较高的前端应用。

本系统前端基于Vue 3进行开发，覆盖登录认证、设备管理、预约管理、借用归还、维修管理、消息中心和统计分析等全部功能页面。通过Vue 3的组件化机制将界面拆分为可复用的视图单元，有效降低了页面间的重复代码比例，提升了整体开发效率与后期维护便利性。

#### 2.6.2 Vite

Vite是面向现代ES模块的前端构建工具，凭借原生ESM（ES Modules）的按需加载机制，在开发阶段实现了极快的冷启动速度和毫秒级的热模块替换（HMR），相较于传统Webpack构建工具在开发体验上具有显著优势。

在本系统中，Vite主要承担前端项目的本地开发服务与生产构建两项职责。其内置的代理配置功能（`/api` → `http://localhost:8081`）可在不引入跨域问题的前提下将前端请求转发至后端服务，有效支撑了前后端分离开发模式下的接口联调工作。

#### 2.6.3 Element Plus

Element Plus是专为Vue 3生态设计的企业级桌面端组件库，提供了表单（Form）、表格（Table）、分页（Pagination）、弹窗（Dialog）、菜单（Menu）、标签（Tag）等70余个常用组件，风格统一、接口规范，适用于管理后台界面的高效构建[12]。

本系统管理端页面全面采用Element Plus进行界面设计，通过其标准组件快速实现设备列表展示（ElTable + 分页）、预约审批弹窗（ElDialog + ElForm）、消息通知列表（ElList + ElBadge）和统计看板（ElCard）等核心界面元素，在有效控制前端开发成本的同时保证了整体界面风格的一致性与专业性。

#### 2.6.4 Vue Router与Pinia

Vue Router是Vue官方提供的客户端路由管理工具，主要用于实现单页应用（SPA）中的页面切换与URL导航。在本系统中，通过Vue Router的全局路由守卫（`router.beforeEach`）对每次页面跳转进行拦截，结合路由元数据（`meta.roles`、`meta.allowFirstLogin`）中声明的权限属性，实现了前端层面的访问控制：未登录用户自动跳转至登录页，权限不足时跳转至403提示页，首次登录未改密时限制访问业务页面。

Pinia是Vue 3官方推荐的状态管理工具，以简洁的Store机制替代了Vuex，适用于维护跨组件共享的应用状态。本系统通过`useAuthStore`管理用户认证状态（令牌、角色、用户信息），认证数据同步持久化至`localStorage`，保证页面刷新后状态正常恢复；通过`useMenuStore`维护当前用户可访问的菜单配置，驱动侧边导航的动态渲染，实现菜单显示与角色权限的实时联动。

### 2.7 数据库与缓存技术

#### 2.7.1 MySQL

MySQL是当前应用最广泛的开源关系型数据库管理系统之一，具有性能稳定、事务支持完善、生态工具丰富等优势，适合承载中小型信息管理系统的结构化数据存储需求。MySQL支持多种存储引擎（默认InnoDB引擎支持行级锁和ACID事务），在索引优化、多表关联查询和并发访问控制方面均具备成熟的技术积累[16]。

本系统选用MySQL 8作为核心持久化存储平台，数据库名称为`lab_booking`，共设计12张业务数据表，涵盖用户与权限、菜单配置、设备管理、预约借还、维修处理、消息通知和任务日志等业务域。通过合理设置唯一约束、外键关联和状态字段枚举，系统在数据库层面对业务数据的一致性与完整性提供了基础保障。在查询性能方面，对高频查询字段（如`login_id`、`device_code`、`status`等）建立了相应索引，有效提升了常用查询操作的响应效率。

#### 2.7.2 Redis

Redis是一种基于内存存储的高性能键值数据库，支持字符串、哈希、列表、集合等多种数据结构，具有读写延迟极低、支持数据过期机制的特点，在缓存管理、会话存储和热点数据加速等场景中应用广泛。

本系统在默认配置下采用基于内存的令牌存储方案（`InMemoryTokenStore`），在单节点部署场景下运行稳定。同时，系统在架构设计上预留了Redis集成扩展接口，通过环境变量`APP_REDIS_ENABLED`控制切换至`RedisTokenStore`实现，以满足多节点水平扩展部署时令牌共享的需要。此外，统计分析模块对高频访问的概览数据引入了缓存机制，可在Redis启用时将查询结果缓存，有效减轻数据库重复计算压力，提升系统在高并发访问下的响应性能。

## 第3章 系统需求分析

### 3.1 总体需求分析

#### 3.1.1 建设背景

高校实验室设备是支撑教学实验、课程实践和科研活动的重要物质基础，其管理水平直接影响实验教学的质量和设备资源的利用效率。有研究指出，教学设备管理信息化的水平已成为衡量高校教学保障能力的重要维度之一，其管理规范性直接关系到设备的使用寿命和教学服务质量[15]。随着高校信息化建设的持续深入推进，传统依赖人工登记或纸质审批的设备管理方式已难以适应现代实验室管理的实际需求，信息化改造的必要性日益凸显[1]。

目前，高校实验室设备管理中普遍存在以下问题：一是设备使用信息更新不及时，师生难以实时掌握设备可用状态，容易出现重复预约或资源闲置现象；二是预约、审批和借还流程较为分散，管理人员难以对相关信息进行统一汇总和全过程跟踪；三是设备借用记录不完整，发生损坏或丢失时责任追溯难度较大；四是维修申请与处理过程缺乏统一平台支撑，维护效率较低，影响设备的正常周转。

针对上述问题，设计并实现一套功能完整的高校实验室设备预约管理系统具有较强的现实意义。通过信息化手段将设备信息维护、预约申请、审批流程、借还记录、维修处理和消息提醒等业务统一纳入系统管理，不仅能够有效提高实验室设备的利用率和流转效率，还能推动实验室管理工作向规范化、数字化方向持续演进，为高校"双一流"建设提供信息化保障[2]。

#### 3.1.2 系统总体目标

本系统面向高校实验室设备管理业务，旨在构建一套功能较为完整、流程较为规范、权限边界清晰的设备预约管理平台。其总体目标主要包括以下几个方面：

（1）构建覆盖超级管理员、管理员、教师和学生四类角色的统一管理平台，实现不同角色职责的合理划分；
（2）实现设备基础信息的统一维护、分类管理和状态跟踪，提高设备信息管理的规范性；
（3）实现预约申请、审批处理、借用登记、归还确认等业务流程的系统化管理，形成较完整的业务闭环；
（4）建立消息通知机制，对预约失效、即将到期、逾期未还及维修处理结果等关键事件进行提醒；
（5）提供基础统计分析功能，为实验室设备配置、使用监控和管理决策提供数据支持；
（6）保证系统具备一定的安全性、可维护性和可扩展性，以满足高校实验室日常管理需求。

#### 3.1.3 系统角色分析

根据系统业务特点与实验室管理实际需求，本系统设定四类用户角色，分别为超级管理员、管理员、教师和学生。不同角色在系统中承担的职责存在明显差异，其权限划分如表3-1所示。

**表3-1 系统角色及职责划分**

| 角色       | 角色代码    | 主要职责                                                       |
| ---------- | ----------- | -------------------------------------------------------------- |
| 超级管理员 | SUPER_ADMIN | 负责系统全局管理，包括管理员账号管理、角色权限配置、菜单管理等 |
| 管理员     | ADMIN       | 负责设备管理、预约终审、借用归还处理、维修管理及统计分析       |

### 3.2 功能需求分析

#### 3.2.1 用户认证与账户管理需求

用户认证与账户管理是系统安全运行的基础模块，直接决定了系统能否对访问行为进行有效管控。系统面向超级管理员、管理员、教师和学生四类角色，不同角色拥有各自对应的身份标识，超级管理员和管理员通过编号或账号登录，教师通过工号登录，学生通过学号登录。系统应在用户提交登录信息后对身份标识、密码和账户状态进行逐级校验，校验通过后分配认证令牌并引导用户进入对应功能界面，校验失败时给出明确的错误提示，拒绝非法访问。

为提升账户使用安全性，系统应设计首次登录强制修改密码机制。新建账号的默认密码统一为初始值，用户首次登录成功后，系统应自动识别其首次登录状态，限制其访问业务功能页面，仅开放个人中心和密码修改入口，待用户完成密码修改并满足最低长度要求后，系统解除限制并允许正常使用全部功能。这一机制能够有效防止默认密码长期存在所带来的安全隐患。

在账户管理方面，系统应按照组织层级设计分层管理能力。超级管理员可创建、启用、禁用管理员账号并重置其密码；管理员可对其管辖的教师账号执行相同操作；教师则可创建、删除和管理其负责的学生账号。这种层级化的账户管理设计能够清晰反映高校实验室的组织结构，防止越权操作。此外，所有用户均应能够在个人中心查看自身基本信息，包括姓名、账号、角色、信用分和历史借用记录，为用户提供统一的个人信息管理入口。

#### 3.2.2 角色与权限管理需求

系统面向多类用户群体，不同角色在功能使用范围和数据访问范围方面存在明显差异，因此需要建立清晰、可配置的权限管理机制，确保每类用户的操作行为始终限定在其职责范围之内。系统应支持对角色所拥有的功能权限进行查询、分配和调整，权限粒度应覆盖主要业务操作，例如设备新增、预约审批、维修处理等，使管理员能够灵活控制各角色的功能边界。

在菜单可见范围方面，系统应根据当前登录用户的角色动态渲染侧边导航菜单，仅展示该角色有权访问的功能页面入口，隐藏无权访问的菜单项，从而降低用户因误操作进入无关页面的概率。菜单与权限应保持独立配置，菜单可见不等于自动拥有对应功能的操作权限，两者需分别进行分配和管理。

超级管理员应能够通过角色权限配置页面查看系统中所有角色的当前权限集合，并根据实际管理需求对权限项和菜单项进行调整。角色权限的变更应在下次请求时生效，以保证权限策略能够及时落地。通过构建基于角色的访问控制体系，系统能够在保证使用便利性的同时，有效防止越权访问和数据泄露风险。

#### 3.2.3 设备管理需求

设备管理是本系统的核心基础功能，其信息质量直接影响预约、借用和维修等业务模块的正常运转。系统应支持对实验室设备的基础信息进行统一维护，管理范围包括设备名称、设备编号、所属类别、当前状态、存放位置、设备图片和描述说明等字段。管理员可对设备信息进行新增、编辑和停用操作，设备编号应具备唯一性约束，防止重复录入。对于设备图片，系统应支持文件上传方式录入，并在设备详情页面提供可视化展示，增强设备信息的直观性。

设备状态管理是该模块的核心能力之一。设备在系统中的状态并非静态不变，而是随预约、借用、归还和维修等业务流程的推进而动态变化。系统应维护完整的设备状态流转机制：设备初始状态为可用，预约审批通过后转为已预约，用户取用后转为借出中，归还且完好时恢复为可用，归还时存在损坏则转为损坏状态，进入维修流程后转为维修中，维修完成后再次恢复可用。通过状态的实时联动更新，系统能够准确反映每台设备的当前实际状态，避免因信息滞后导致的设备冲突或重复预约问题。

在设备检索方面，系统应支持按设备名称关键字、设备编号关键字、类别和状态等多条件组合筛选，并以分页方式返回查询结果，避免一次性加载大量数据影响页面响应速度。管理员可通过筛选功能快速定位目标设备并执行相应操作，学生和教师则可通过设备列表查询当前可预约的设备信息，提高设备资源的利用效率。

#### 3.2.4 预约管理需求

预约管理是系统业务流程的起始环节，其设计质量直接决定了设备资源能否被合理分配和高效利用。系统应支持学生和教师用户在确认设备当前可用后提交预约申请，申请内容包括目标设备、预约开始时间、预约结束时间和使用目的。为保证同一设备不被重复占用，系统应在接受预约申请前执行时间冲突检测，对于与已有有效预约存在时间段重叠的申请，系统应明确拒绝并提示冲突原因，引导用户调整预约时间。

在审批流程设计上，系统应根据申请人的角色实施分层审批机制，以适应高校实验室管理的实际组织结构。学生提交的预约申请需先由所属教师进行初步审核，教师审核通过后方可进入管理员终审环节；而教师提交的预约申请则可直接进入管理员审批阶段，减少不必要的审批层级。审批操作应支持通过和驳回两种结果，驳回时要求填写审批意见，以便申请人了解被驳回的原因并做出调整。

终审通过后，系统应自动生成对应的借用记录，并将设备状态更新为已预约，为后续取用操作做好数据准备。预约申请在提交后、进入最终执行状态前，申请人应可主动取消，取消后设备状态恢复为可用。此外，系统应设计预约自动失效机制，对于终审通过后超过规定时间仍未被取用的预约记录，系统通过定时任务将其标记为已失效，释放设备资源，同时向申请人推送失效提醒，避免设备长期被无效占用。

#### 3.2.5 借用归还管理需求

借用归还管理是预约审批完成后的执行环节，负责记录设备从实际取用到最终归还的完整过程，是保障设备流转可追溯的重要机制。系统应在预约终审通过后自动生成借用记录，记录内容包括借用人、借用设备、关联预约、预计归还时间等基础信息，初始状态为待取用。管理员在用户实际领取设备时应在系统中执行取用确认操作，系统同步记录实际取用时间并将设备状态更新为借出中，实现线下行为与系统数据的一致性。

在设备归还环节，管理员应在用户归还设备时录入归还时间和设备状况说明。若设备归还时状态完好，系统将借用记录标记为已归还，并将设备状态恢复为可用；若归还时设备存在损坏，系统在更新借用记录状态的同时将设备标记为损坏状态，并可自动触发维修申请流程，确保损坏设备的处理不被遗漏。借用记录的设备状况字段为后续责任追溯提供了依据，有助于规范用户的借用行为。

系统还应具备逾期自动识别能力。当借用记录的预计归还时间到达后仍未办理归还手续，系统通过定时扫描任务将相应记录状态更新为逾期，并同步向借用人发送逾期提醒通知，督促其及时归还设备。对于临近归还时间但尚未逾期的借用记录，系统还应提前生成到期提醒，帮助借用人合理安排归还时间，从整体上提高设备的流转效率和归还及时率。

#### 3.2.6 消息通知需求

消息通知模块是提升系统信息传递效率的重要组成部分，通过在关键业务节点向用户推送系统消息，可以有效减少因信息滞后而导致的管理问题。系统应在以下场景自动生成并推送相应通知：用户首次登录时推送密码修改提醒；管理员重置某用户密码后推送密码重置通知；预约记录因未及时取用而自动失效时推送失效提醒；借用记录临近到期时推送到期提醒；借用记录超期后推送逾期提醒；维修申请状态发生变更时向申请人推送处理结果通知。这些通知覆盖了系统主要业务链路的关键节点，能够让用户及时感知与自身相关的业务进展。

系统应为用户提供统一的消息中心入口，集中展示所有推送给当前用户的通知信息，按时间倒序排列，最新通知优先展示。消息应分为未读和已读两种状态，用户可在消息中心对单条消息进行已读确认，系统记录确认时间。导航栏应展示当前用户未读消息的数量角标，使用户无需进入消息中心也能感知到待处理的通知信息，提高消息的触达效果。

为防止相同事件重复推送干扰用户，系统应在生成通知时判断是否已存在相同类型且关联相同业务对象的未读通知，若已存在则跳过本次推送，保证消息的不重复性。消息内容应包含清晰的标题、正文说明和关联业务类型，使用户能够快速理解通知含义并采取相应行动。

#### 3.2.7 维修管理需求

当设备在借用或日常使用过程中发生故障或损坏时，系统应提供规范化的维修申请和处理流程，确保设备损坏信息得到及时记录和妥善处置。用户在发现设备损坏后应能够在系统中提交维修申请，填写内容包括损坏设备和故障描述，系统接收申请后将设备状态更新为维修中，防止其在维修期间被其他用户预约或借用。为避免同一设备被重复提交维修申请，系统应在接受新申请前检查目标设备是否已存在处理中的维修记录，若有则拒绝重复提交并给出提示。

在维修处理环节，管理员应能够对维修记录进行全程跟踪管理。管理员可将维修状态由待处理更新为处理中，表示维修工作已经启动；维修完成后将状态更新为已完成，系统同步将设备状态恢复为可用；若设备经评估无法修复，则将维修状态标记为无法修复，设备维持损坏状态，管理员可根据实际情况决定后续处置方式，如停用或报废。每次状态变更时，管理员应填写处理说明，使申请人能够了解维修的具体进展和结论。

维修状态变更完成后，系统应向维修申请人自动推送通知，告知其维修结果，增强维修流程的透明性和用户体验。通过建立从损坏报告、维修处理到结果反馈的完整管理闭环，系统能够规范实验室设备的维护行为，减少设备损坏信息遗漏的风险，提高设备的整体维护水平和使用寿命。

#### 3.2.8 统计分析需求

统计分析模块为管理人员提供数据支撑，帮助其全面掌握系统运行状态和设备资源使用情况，从而为实验室管理决策提供参考依据。系统应提供业务数据概览功能，将设备总量、当前可用设备数、借用中设备数、预约总量、借用总量和维修总量等核心指标以数据卡片形式集中展示，使管理人员无需逐一翻阅业务列表即可快速掌握系统整体运行情况，在登录首页直接获取关键数据。

在排行榜统计方面，系统应提供热门设备榜、设备损坏榜和用户违规榜三类统计功能，并支持按累计、近半年和近一个月三种时间维度切换查看。热门设备榜通过统计各设备的借用次数，帮助管理人员识别使用频率最高的设备，为合理安排设备维护计划和采购优先级提供参考。设备损坏榜通过汇总各设备的维修申请次数，识别故障率较高的设备，辅助管理人员评估设备使用寿命和维保需求。用户违规榜通过统计各用户的逾期归还次数和损坏设备次数，帮助管理人员识别借用行为不规范的用户，作为信用管理和行为约束的数据依据。

管理人员还应能够灵活控制排行榜的展示数量，支持选择前3、前5或前10名进行展示，以适应不同场景下的数据查看需求。各统计数据应基于系统历史业务记录实时计算生成，保证统计结果的准确性和时效性。通过这一系列统计分析功能，系统能够将分散的业务数据转化为有价值的管理洞察，帮助实验室管理人员更科学地开展设备资源配置、使用监控和制度优化工作。

### 3.3 非功能需求分析

非功能需求从质量属性角度约束系统的构建方式，是保障系统在实际运行中可用、可信、可维护的重要依据。本系统的非功能需求主要涵盖安全性、数据一致性、可用性与性能、可扩展性和可维护性五个维度。

#### 3.3.1 安全性需求

系统应构建多层次的安全防护体系。在访问控制层面，应通过完善的身份认证机制确保只有持有有效令牌的合法用户方可访问受保护资源；通过基于RBAC模型的角色权限控制，将用户的可操作范围严格限定在其职责范围内，防止越权访问与数据泄露[11]。在密码安全层面，用户密码应经过BCrypt等强哈希算法加密后存储，避免明文密码因数据库泄露而带来的安全风险；首次登录强制修改密码机制可有效规避默认密码长期存在的安全隐患。在接口防护层面，所有业务接口应在请求过滤层完成令牌合法性校验，对未携带令牌或令牌失效的请求统一返回401错误码，拒绝非法访问。

#### 3.3.2 数据一致性需求

在预约审批、借用归还、维修处理等核心业务操作中，往往涉及多张业务数据表的联动更新（如设备状态表、借用记录表、维修记录表的同步变更）。为防止部分操作成功、部分操作失败导致的数据不一致状态，系统应对关键业务操作采用数据库事务进行原子化控制，确保业务操作要么全部成功、要么全部回滚[16]。同时，通过在数据库表中设置必要的唯一约束、非空约束和状态枚举约束，在数据库层面为业务数据的完整性提供额外保障。

#### 3.3.3 可用性与性能需求

系统应具备满足实验室日常管理工作连续使用需求的可用性。在功能层面，对设备列表、预约记录、借用记录等高频访问接口应支持分页查询，避免一次性加载全量数据引发界面响应迟缓。在中小规模高校实验室应用场景下（设备数量约50至200件、用户规模约30至100人），系统核心接口的响应时间应控制在500毫秒以内，以满足基本的交互流畅性要求。在系统稳定性方面，应通过统一的异常处理机制防止未处理异常导致服务崩溃，保证系统具有较好的健壮性。

#### 3.3.4 可扩展性需求

随着实验室管理需求的演进，系统在后续使用过程中可能面临新增角色类型、扩展业务流程、接入物联网设备或开发移动端应用等扩展需求[2]。因此，系统在设计阶段应充分遵循模块解耦原则，各功能模块之间通过清晰的接口边界进行交互，避免直接依赖内部实现；后端RESTful接口应设计为无状态服务，便于后续水平扩展；前端组件化架构应支持按需增减功能页面而不影响整体结构。

#### 3.3.5 可维护性需求

系统在代码组织层面应严格遵循分层架构思想，保持控制层、业务层和数据访问层之间的职责边界清晰，降低跨层依赖带来的维护复杂度。接口设计应遵循统一规范（统一响应格式、统一错误码体系），便于前后端协作调试与问题定位。数据库表结构应保持良好的命名规范和注释说明，以降低数据层的理解成本。此外，系统部署配置项（数据库连接、Redis开关等）应通过环境变量进行外部化管理，支持在不修改代码的情况下灵活调整运行参数，提升系统在不同环境下的部署适应能力。

# 第4章 系统设计

系统设计是需求分析向代码实现过渡的关键环节，主要工作包括系统整体架构的确立、各功能模块的划分与接口定义，以及数据库表结构的设计。良好的系统设计能够从顶层确立各模块之间的职责边界与协作方式，为后续实现工作提供清晰、一致的工程蓝图，也是保障系统可维护性与可扩展性的重要基础。

## 4.1 系统架构设计

本系统采用前后端分离的B/S（Browser/Server）架构模式。用户通过浏览器访问系统前端界面，前端负责页面展示、用户交互与接口调用；后端负责业务逻辑处理、权限控制与数据持久化；数据库承担系统业务数据的结构化存储。

在整体技术选型上，前端基于Vue 3构建单页应用（SPA），后端基于Spring Boot整合Spring MVC、Spring Security和MyBatis构建RESTful服务接口，前后端之间通过HTTP协议通信，数据交换格式统一采用JSON。前后端分离架构具有职责划分清晰、开发可并行推进、前端可独立部署迭代等优点[17][19]，适合高校实验室设备预约管理系统这类具有明确角色体系与多模块业务的管理信息系统建设场景。

从后端内部结构来看，系统遵循典型的三层分层设计：控制层（Controller）负责接收HTTP请求、完成参数校验并组织响应数据；业务层（Service）负责实现系统核心业务逻辑，包括预约审批流转、借用归还处理、权限校验和设备状态联动等；数据访问层（Mapper）负责通过MyBatis与MySQL数据库进行交互，完成业务数据的读写操作。三层之间通过接口定义进行依赖，控制层依赖业务层接口，业务层依赖数据访问层接口，各层内部实现细节对上层透明，有效降低了模块间的耦合度，便于后续功能扩展与单元测试。

在安全控制层面，系统通过Spring Security的过滤链机制统一处理认证与授权逻辑，在每次请求到达控制层之前完成令牌解析与身份提取；在数据存储层面，系统以MySQL作为核心持久化数据库，并在架构上预留了Redis扩展能力，以满足后续会话共享和缓存加速的需要。

## 4.2 功能模块设计

根据系统需求分析结果，结合高校实验室设备管理的业务特点，系统主要划分为认证登录、用户管理、角色权限管理、设备管理、预约管理、借用归还管理、维修管理、消息通知管理、统计分析和菜单配置等功能模块。

认证登录模块主要负责用户身份认证、登录状态维护和密码修改等功能，是系统运行的基础模块。用户管理模块主要用于实现不同角色用户的创建、启用、禁用和信息维护。角色权限管理模块用于维护系统角色对应的功能权限和菜单可见范围，从而确保不同用户只能访问其职责范围内的资源。

设备管理模块主要用于维护实验室设备的基础信息，包括设备名称、编号、类别、状态、位置和描述等内容，同时支持设备信息查询、编辑和状态调整。预约管理模块是系统的核心模块之一，主要负责设备预约申请、审批流转、预约取消和预约失效处理等业务。借用归还管理模块在预约审批通过后记录设备借用情况，并负责归还登记和逾期管理。维修管理模块用于处理设备损坏后的报修、维修状态更新及维修结果维护。消息通知模块用于向用户推送预约失效、归还提醒、逾期通知和维修结果等业务消息。统计分析模块则通过对设备、预约、借用和维修等数据进行汇总分析，为管理人员提供数据支持。

各模块之间并非相互独立，而是存在密切的业务关联。例如，预约管理依赖设备管理模块提供设备状态信息，借用归还管理依赖预约审批结果生成借用记录，维修管理又与借还过程中的设备损坏情况相关联，消息通知模块则贯穿多个业务模块的关键节点。通过这种模块化设计，系统在保持结构清晰的同时，也能够较好地支撑完整的实验室设备管理业务流程。

## 4.3 界面设计

本系统前端界面面向多角色用户，在兼顾不同角色操作需求差异的同时，遵循一致性、可用性和简洁性原则进行设计，基于Element Plus组件库构建统一的视觉风格[12]。

### 4.3.1 整体布局设计

本系统前端采用典型的三栏式后台管理布局：左侧为固定导航侧边栏，顶部为操作栏，右侧为主内容区。左侧侧边栏根据当前登录用户的角色，从后端获取可访问菜单配置后动态渲染，不同角色用户看到的导航菜单项不同，超出权限范围的菜单项对用户不可见，从交互层面落实了前端访问控制。顶部操作栏展示系统名称、当前用户姓名、角色标识及未读消息角标计数，点击消息角标可快速跳转至消息中心；右上角提供退出登录入口。主内容区通过`<RouterView>`加载各功能页面组件，页面切换时保持侧边栏与顶栏稳定，减少用户操作的认知负担。

该三栏式布局结构清晰、操作路径明确，在多模块切换和多角色差异化展示场景下均具有良好的可用性，符合后台管理系统的主流交互规范[12]。

### 4.3.2 登录页面设计

登录页面是所有用户进入系统的统一入口，界面以简洁为原则，主要由系统标题、登录标识输入框（支持学号、工号或管理员编号）、密码输入框和登录按钮构成。不同角色用户通过同一登录入口提交凭证，后端根据`login_id`或`account`字段匹配用户信息并完成校验，校验通过后返回令牌和用户角色，前端据此跳转至对应角色的默认功能页面。

首次登录时，若后端返回`firstLoginRequired=true`标记，前端通过`ElMessage.warning`弹出醒目提示，并在路由守卫层限制用户访问业务功能页面，仅开放个人中心与密码修改入口，强制引导用户完成密码修改后方可正常使用系统。该设计在不增加页面复杂度的前提下有效落实了账户安全管控要求。

### 4.3.3 设备管理页面设计

设备管理页面承担实验室设备台账的日常维护功能，页面整体分为查询筛选区和数据列表区两部分。筛选区支持按设备名称关键字、设备编号关键字、所属类别和当前状态进行多条件组合查询，条件变化时前端自动重新拉取数据；数据列表区通过`ElTable`组件以分页表格形式展示设备编号、名称、类别、当前状态、位置及操作按钮，其中状态字段通过`ElTag`组件以不同颜色区分（可用为绿色、借出中为橙色、维修中为蓝色、损坏为红色、停用为灰色），增强信息辨识度。

新增和编辑操作均通过`ElDialog`弹窗表单实现，避免页面跳转打断当前视图状态；表单提交前通过`ElForm`完成必填项校验和业务规则验证（如设备编号唯一性提示）。设备图片支持通过URL输入或文件上传方式录入，并在详情弹窗中提供可视化预览，增强设备信息的直观性[9]。

### 4.3.4 预约管理页面设计

预约管理页面根据当前用户角色呈现差异化的内容与操作。学生用户界面以个人预约申请表单和历史预约记录列表为核心，支持查询当前可预约设备并提交新申请，系统在申请时间段与已有预约冲突时前端给出明确提示；教师用户在个人预约记录基础上额外展示待初审的学生预约列表，可对每条记录执行通过或驳回操作，驳回时需填写审批意见；管理员用户则显示待终审预约队列，执行最终审批决策。

预约详情弹窗完整展示申请人信息、申请设备、预约时间段、使用目的和各环节审批意见，便于审批人全面了解申请背景后做出判断。整体设计遵循角色最小权限原则，每类角色仅能看到与其业务职责直接相关的视图内容，避免界面信息过载[11]。

### 4.3.5 借用归还页面设计

借用归还页面面向管理员，是线下设备交接行为的数字化记录入口。页面核心为借用记录列表，支持按借用状态（待取用、借用中、已归还、逾期）进行筛选定位；在对应记录行提供"确认取用"和"登记归还"两类操作按钮，按钮依据当前借用状态的合法性动态启用，防止误操作触发不合法的状态变更。

归还登记弹窗要求填写设备归还时的实际状况（完好/损坏），若填写为"损坏"，系统在提交后自动将设备状态更新为`DAMAGED`并创建维修申请，用户无需二次操作。该联动设计减少了管理员的重复操作成本，同时保证了设备损坏信息不被遗漏，增强了业务流程的健壮性[9]。

### 4.3.6 消息中心页面设计

消息中心以统一入口集中展示系统向当前用户推送的所有通知，消息按创建时间倒序排列，最新通知优先呈现。每条消息包含消息标题、正文内容、消息类型标签（预约提醒、逾期提醒、密码通知、维修结果等）及创建时间；未读消息以视觉差异（加粗或背景色）与已读消息区分，用户点击确认后状态更新为已读，系统记录确认时间。

顶部操作栏中的消息图标实时显示未读消息数量角标，用户无需主动进入消息中心即可感知到待处理通知。角标数据在每次页面加载和关键业务操作完成后刷新，保持与实际未读数量的一致性，有效提升了消息的可见性与触达效率。

### 4.3.7 统计分析页面设计

统计分析页面专供管理员使用，通过数据可视化手段将系统运行状态以直观方式呈现，辅助管理层进行资源配置决策[7]。页面分为概览区和排行榜区两个层次。

概览区采用`ElCard`数据卡片网格形式，集中展示设备总量、当前可用设备数、借用中设备数、预约总量、借用总量和维修总量六项核心指标，每项指标对应独立卡片，数值突出展示，管理员在系统主页即可快速掌握整体运行状况。

排行榜区提供热门设备榜（按借用次数排序）、设备损坏榜（按维修申请次数排序）和用户违规榜（按逾期与损坏次数排序）三类统计，均支持按累计、近半年和近一个月三种时间维度切换，以及前3、前5或前10项展示数量的灵活配置，适应不同管理场景的数据查看需求。排行榜以表格结合序号标识的形式呈现，兼顾信息密度与阅读清晰度。

## 4.4 数据模型设计

### 4.4.1 概念模型设计

为保证系统业务数据能够被规范存储和高效管理，本系统采用关系型数据库进行数据建模，数据库名称为`lab_booking`。在进行物理表结构设计之前，系统先行完成了概念层面的实体关系（E-R）分析，明确各业务实体的核心属性及其相互关联方式，为后续表结构的细化设计提供逻辑依据。

根据系统业务需求，数据模型中识别出以下主要业务实体：

**用户（User）**：系统最核心的基础实体，代表系统中的各类操作主体，包含姓名、登录标识、角色编码、账户状态、信用积分和密码等核心属性。用户实体通过上级管理者字段（`manager_id`）实现内部层级自关联，体现超级管理员—管理员—教师—学生的层级组织结构。

**角色（Role）**与**权限（Permission）**：角色实体定义系统中存在的角色类型（超级管理员、管理员、教师、学生），权限实体定义各项操作权限的编码标识（如`device:create`、`reservation:approve`）。用户与角色之间存在多对一关系，角色与权限之间存在多对多关系，通过`sys_role_permission`关联表实现，共同构建RBAC权限控制模型。

**菜单（Menu）**：代表前端侧边栏的导航菜单项，包含菜单名称、前端路由路径和图标等属性。菜单与角色之间存在多对多关系，通过`sys_role_menu`关联表控制不同角色的可见菜单范围，实现前端界面元素的动态权限配置。

**设备（Device）**：代表实验室中可供预约和借用的物理设备，包含设备名称、设备编号（唯一约束）、类别、状态、存放位置和图片路径等属性。设备状态字段在业务流程推进中动态变化，是连接预约、借用和维修三类业务实体的核心枢纽。

**预约记录（Reservation）**：代表用户对设备发起的预约申请，包含预约时间段、使用目的、审批意见和流程状态等属性。预约记录与用户实体存在多对一关系（一个用户可有多条预约记录），与设备实体存在多对一关系（一台设备可有多条历史预约记录）。预约状态字段随审批流程逐步流转（`PENDING→APPROVED→PICKUP_PENDING→EXPIRED/CANCELLED`）。

**借用记录（BorrowRecord）**：代表预约审批通过后的实际借用行为，包含取用时间、预计归还时间、实际归还时间和归还时设备状况等属性。借用记录与预约记录之间存在一对一关系，与用户、设备实体均存在多对一关联，是记录设备流转轨迹的核心载体。

**维修记录（Repair）**：代表设备损坏后的报修事件，包含故障描述、处理状态（`PENDING/PROCESSING/COMPLETED/UNREPAIRABLE`）和处理说明等属性。维修记录与设备实体存在多对一关系（一台设备可有多次维修历史），与用户实体存在多对一关系（记录申请人信息）。

**消息通知（Notification）**：代表系统向用户推送的业务提醒，通过`type`字段区分不同业务场景下的通知类型，并通过`related_biz_type`和`related_biz_id`字段关联触发通知的具体业务记录。消息通知与用户实体存在多对一关系（一个用户可有多条通知）。

从整体数据关系来看，用户实体（User）与预约、借用、维修和通知四类业务实体均存在一对多关联；设备实体（Device）分别与预约、借用和维修实体形成一对多关联；预约与借用实体之间存在严格的一对一约束关系，共同构成系统核心业务流的数据骨架。通过主键约束、唯一约束、外键关联及状态字段枚举等数据库级别的完整性保障机制，系统能够支持设备从预约申请、借用使用到归还维修的完整生命周期数据管理，为业务流程的可靠执行提供坚实的数据模型基础。

### 4.4.2 数据库表设计

根据系统功能模块划分，数据库`lab_booking`共包含12张数据表，涵盖用户与权限、菜单配置、设备管理、预约借还、维修处理、消息通知和任务日志等业务域。以下对各表结构进行详细说明。

**（1）sys_user（用户表）**

用户表用于存储系统中所有用户的基本信息、身份认证信息和账户状态，是系统认证与权限控制的核心基础表。其中`login_id`字段用于区分不同角色用户的登录标识（如学号、工号或管理员编号），`manager_id`字段用于记录上级管理者关系，`first_login_required`字段用于标记用户是否需要强制修改初始密码。

| -------------------- | ------------ | ---------------- | --------------------------------------------- |
| user_id              | BIGINT       | 主键             | 用户唯一标识                                  |
| name                 | VARCHAR(64)  | NOT NULL         | 用户真实姓名                                  |
| account              | VARCHAR(64)  | UNIQUE           | 系统账号（可选）                              |
| login_id             | VARCHAR(64)  | UNIQUE, NOT NULL | 登录标识（学号/工号/编号）                    |
| phone                | VARCHAR(20)  |                  | 手机号码                                      |
| role_code            | VARCHAR(32)  | NOT NULL         | 角色编码（SUPER_ADMIN/ADMIN/TEACHER/STUDENT） |

权限表用于存储系统中各功能操作对应的权限标识，是接口级细粒度权限控制的基础数据，每条记录对应一项具体的操作权限（如设备新增、预约审批等）。

**表4-3 sys_permission 权限表**

菜单表用于存储系统功能菜单项信息，包括菜单名称、前端路由路径、图标及关联权限编码，与角色通过关联表共同决定不同角色的可见菜单范围。

## 4.5 接口设计

### 4.5.1 接口规范设计

系统采用前后端分离架构，后端需向前端提供统一规范的接口服务。本系统后端接口整体遵循RESTful设计风格[17]，以资源为中心组织接口路径，以HTTP请求方法区分操作类型：GET用于数据查询，POST用于数据新增，PUT用于数据更新，DELETE用于数据删除。所有接口的基础路径统一为`/api`，数据传输格式统一采用JSON，时间字段统一使用`yyyy-MM-dd HH:mm:ss`格式。

**（1）认证方式**

系统采用基于令牌的无状态认证机制。用户登录成功后，后端返回令牌字符串，客户端在后续每次请求中通过HTTP请求头`Authorization: Bearer <token>`携带令牌。后端认证过滤器从请求头中提取令牌并完成身份校验，校验通过后将用户信息写入安全上下文，供后续接口处理使用。

**（2）统一响应结构**

所有接口返回统一的JSON结构，包含业务状态码`code`、提示信息`message`和业务数据`data`三个字段。其中`code`为0时表示操作成功，非0时表示业务异常。对于分页查询接口，`data`字段中包含`list`（数据列表）、`pageNum`（当前页码）、`pageSize`（每页条数）和`total`（总记录数）四个固定字段。响应结构示例如下：

```json
{
    "code": 0,
    "message": "success",
    "data": {
        "list": [],
        "pageNum": 1,
        "pageSize": 10,
        "total": 100
    }
}
```

**（3）异常响应机制**

接口层设计了统一的全局异常处理机制。对于常见异常场景，系统返回对应的HTTP状态码与业务错误码：参数校验失败返回400，未认证访问返回401，权限不足返回403，资源不存在返回404，业务状态冲突（如时间冲突、状态非法）返回409。通过统一的错误响应结构，前端能够快速定位问题原因，提升联调效率。

# 第5章 系统模块详细设计与实现

本章依据第4章完成的系统设计，对各核心功能模块的后端逻辑与前端实现进行详细阐述。每个模块分别从后端设计与实现、前端设计与实现两个维度展开，重点说明关键业务逻辑的实现思路、核心数据流转机制以及主要技术处理方案，并结合典型代码片段加以说明。

## 5.1 项目结构设计

为提高代码组织的规范性和可维护性，本系统采用前后端分离的工程结构，后端与前端分别维护独立的项目目录（`admin-server/`与`admin-ui/`），通过RESTful接口进行数据交互，互不耦合。

后端项目（`admin-server`）基于Maven管理依赖，按照功能职责分层组织源码目录：`controller/`层负责接口请求的接收与响应；`service/`层负责核心业务逻辑处理；`repository/`层（Mapper）负责数据库操作；`model/`层存放实体类、请求体和响应体等数据对象；`config/`层统一管理Spring Security配置、静态资源映射、定时任务注册等横切关注点。分层结构使各模块的职责边界清晰，便于后续功能扩展和问题定位。

前端项目（`admin-ui`）基于Vite和Vue 3组织工程结构：`views/`目录存放各业务功能页面视图组件；`router/`目录定义路由配置与路由守卫逻辑；`stores/`目录实现基于Pinia的状态管理（认证状态、菜单状态）；`components/`目录存放跨页面可复用的公共组件；`api/`目录集中封装与后端接口的HTTP请求函数，统一处理请求拦截与响应错误。该结构能够较好地支撑中后台管理系统的组件化开发模式，降低各功能页面之间的代码重复，提升整体可维护性。

## 5.2 认证模块设计与实现

认证模块是系统安全运行的基础，主要负责用户登录、身份识别、登录状态维护及密码修改等功能。系统采用基于令牌的无状态认证方式，结合Spring Security过滤链完成请求级别的身份校验，并通过首次登录强制改密机制提升账户安全性。

### 5.2.1 后端设计与实现

后端认证功能由`AuthController`、`AuthService`和`AuthTokenFilter`三个核心组件协同完成。`AuthController`对外暴露`POST /api/auth/login`、`GET /api/auth/me`和`PUT /api/auth/password`三个接口，接收请求并将处理委托给`AuthService`。

`AuthService`中的`login`方法负责完整的登录校验流程：首先通过`AuthRepository`根据`loginId`或`account`查询用户信息，若账号不存在则直接返回401错误；其次比对密码明文，密码错误同样返回401；再次检查账户的`status`字段，若为`DISABLED`则返回403拒绝访问。三项校验全部通过后，调用`AuthRepository.createToken`生成唯一令牌字符串，将令牌与用户ID的映射关系存入`InMemoryTokenStore`（或在Redis启用时切换至`RedisTokenStore`），最终将令牌、过期时间、首次登录标记及用户基本信息组装为JSON返回给客户端。

请求过滤依赖`AuthTokenFilter`，该类继承自Spring的`OncePerRequestFilter`，在每次请求到达业务控制器之前执行一次身份提取逻辑。过滤器从请求头`Authorization`字段中截取`Bearer`后的令牌字符串，通过`AuthRepository.findByToken`查找关联用户，若命中则将用户对象封装为`UsernamePasswordAuthenticationToken`并写入`SecurityContextHolder`，使后续所有业务层均可通过`authService.currentUser()`直接获取当前登录用户。这种基于令牌的无状态认证模式可有效避免服务端维护会话带来的状态同步问题，在Web应用的安全设计中具有广泛应用[18]。

首次登录强制改密机制也在`AuthTokenFilter`中实现。当安全上下文中的用户`firstLoginRequired`字段为`true`，且请求路径不属于白名单（`/api/auth/me`和`/api/auth/password`）时，过滤器直接截断请求，返回`code:403`的业务错误响应，阻止用户访问其他功能接口。`changePassword`方法在密码修改成功后会将`firstLoginRequired`置为`false`并持久化，使该限制自动解除。

### 5.2.2 前端设计与实现

前端认证功能以`LoginView.vue`为入口界面，使用`useAuthStore`统一管理认证状态。`LoginView.vue`通过Vue 3的`reactive`管理表单数据，结合Element Plus的`ElForm`组件实现必填校验，用户点击登录后调用`authStore.signIn`方法发起登录请求。若服务端返回`firstLoginRequired=true`，页面通过`ElMessage.warning`提醒用户修改密码；登录成功后根据路由中的`redirect`参数跳转至目标页面。

认证状态由`useAuthStore`（基于Pinia实现）统一维护，其内部`state`对象包含`session`（令牌及用户简要信息）、`currentUser`（完整用户信息）和`initialized`（初始化标记）三个字段。`signIn`方法在接口调用成功后，将响应数据同步写入`localStorage`（键名`admin-auth-session`），令牌单独以键名`token`存储，实现跨页面刷新的状态持久化。`signOut`方法调用时清除全部本地存储项并重置`state`。`ensureCurrentUser`方法在路由守卫中被调用，确保每次页面切换时当前用户信息处于最新状态，若接口返回401则自动触发退出流程。

## 5.3 用户管理模块设计与实现

用户管理模块主要用于完成系统内不同角色用户的创建、查询、启用、禁用和密码重置等操作。本系统存在超级管理员、管理员、教师和学生四类角色，模块在实现过程中通过层级校验严格控制操作边界，保证管理权限不被越级使用。

### 5.3.1 后端设计与实现

用户管理后端以`UserController`为入口，将请求分发至`UserService`处理。`UserService.listUsers`方法首先调用`requireRoles`校验当前用户角色，仅允许超级管理员、管理员和教师访问；随后从内存中获取全量用户列表，通过`visibleTo`方法过滤出当前用户有权查看的下级用户，再依据`keyword`、`roleCode`和`status`条件进行二次过滤，最后进行内存分页返回。该方式避免了对数据库的每次全量查询，在中小规模数据量下具有较好的响应性能。

在用户创建方面，系统针对不同角色分别提供`createAdmin`、`createTeacher`和`createStudent`三个方法，各方法内部通过`requireRoles`强制要求调用方必须具备对应权限。新建用户的默认密码统一设为`000000`，且`firstLoginRequired`初始化为`true`，强制首次登录后修改密码。创建管理员账号后，系统同时向目标用户写入一条`FIRST_LOGIN_PASSWORD_CHANGE`类型的消息通知。

账户状态管理由`updateUserStatus`方法完成，执行前同样通过`visibleTo`校验操作目标是否属于当前用户的管辖范围，防止越权操作。密码重置由`resetPassword`方法处理，重置后将密码设为初始值，`firstLoginRequired`重新置为`true`，并通过`NotificationService`向目标用户发送`PASSWORD_RESET`类型的消息通知。删除学生账号采用逻辑删除方式，将`deleted`字段置为`true`，同时校验该学生名下是否存在未归还的借用记录，若存在则拒绝删除。

### 5.3.2 前端设计与实现

前端用户管理页面由`UserManagementView.vue`实现。页面顶部为查询区域，提供关键字、角色和状态三个筛选条件，支持组合搜索；下方为Element Plus的`ElTable`组件展示的分页用户列表，包含用户名、账号、角色、状态和信用分等字段。

页面通过`computed`属性`canManage`和`canCreate`根据当前登录用户的角色动态控制操作按钮的显示与可用状态。新增用户和编辑用户操作均通过`ElDialog`弹窗表单实现，表单项根据创建角色的不同动态调整必填字段。启用/禁用操作通过`ElMessageBox.confirm`二次确认后调用对应接口，密码重置同样通过确认弹窗保护，防止误操作。

## 5.4 设备管理模块设计与实现

设备管理模块是系统核心基础模块之一，主要用于实现实验室设备信息的维护与状态管理。系统支持设备信息的新增、查询、编辑和停用等功能，并通过设备状态与业务流程的联动更新，实时反映设备的实际可用情况。

### 5.4.1 后端设计与实现

设备管理后端由`DeviceController`和`DeviceService`构成。`DeviceService.listDevices`方法通过`DeviceMapper.selectAll`加载全量设备列表，在内存中进行关键字模糊匹配（匹配`deviceName`、`deviceCode`和`location`字段）、类别精确过滤和状态精确过滤，随后按`deviceId`升序排列并完成内存分页，以`list`、`total`、`pageNum`和`pageSize`四个字段组装响应结果。

设备新增和编辑均通过`createDevice`和`updateDevice`方法处理。两者在保存前均调用`DeviceMapper.countByDeviceCode`检查设备编号的唯一性，若与已有记录重复则抛出业务异常。新增设备时，`DeviceService`调用`DeviceMapper.selectNextDeviceId`获取自增ID，保证主键不重复。

设备图片上传由独立的`importDevice`方法（通过`DeviceImportController`暴露）处理，接收`MultipartFile`类型的图片文件，将其保存至配置项`app.storage.upload-dir`指定的本地目录下，以`deviceId`作为文件名，并将相对访问路径写入`lab_device`表的`image_url`字段。静态资源访问通过`StaticResourceConfig`将`/uploads/**`路径映射至本地目录，前端可直接通过URL访问设备图片。

设备状态联动在各业务模块中实现：`ReservationService`在终审通过后将设备状态更新为`RESERVED`；`BorrowRecordService`在确认取用时更新为`BORROWED`；归还且完好时恢复为`AVAILABLE`；归还时损坏则更新为`DAMAGED`；`RepairService`在维修完成后恢复为`AVAILABLE`。通过这种分布式状态联动，设备表始终准确反映设备当前状态。

### 5.4.2 前端设计与实现

前端设备管理页面由`DeviceManagementView.vue`实现。页面通过`computed`属性`canManage`判断当前用户是否为超级管理员或管理员，控制新增、编辑和状态调整按钮的可用性；`canDelete`属性进一步将删除权限限定为超级管理员。筛选区域使用`reactive`管理`filters`对象，监听关键字、类别、状态变化，变化时触发`loadDevices`重新请求设备列表。

设备列表通过`ElTable`展示，状态字段通过`ElTag`组件以不同颜色区分（可用为绿色、借出为橙色、停用为灰色等），增强视觉辨识度。新增和编辑操作共用一个`ElDialog`弹窗表单，通过`saveForm.deviceId`是否为空区分新增和编辑两种模式，表单项包括设备名称、编号、类别、位置、图片URL和描述等字段，提交前通过`ElForm`验证必填项。设备详情通过另一个只读弹窗展示，包含设备图片的预览。

## 5.5 预约管理模块设计与实现

预约管理模块是系统最核心的业务模块，涵盖预约申请、分级审批、主动取消和自动失效等完整流程。模块的核心设计难点在于多角色审批流转与设备状态一致性的协同维护。

### 5.5.1 后端设计与实现

预约管理后端由`ReservationController`和`ReservationService`实现。`createReservation`方法在接收申请时依次执行以下校验：通过`requireRoles`确认申请人为学生或教师；从`DeviceMapper`中查询目标设备并调用`ensureReservable`确认其状态为`AVAILABLE`；解析`startTime`和`endTime`并校验结束时间晚于开始时间；最后调用`ensureNoTimeConflict`方法遍历该设备所有有效预约记录，检测新申请时间段是否与已有记录存在重叠。全部校验通过后，根据申请人角色设置初始状态：教师申请直接设为`APPROVED`，学生申请设为`PENDING`，并持久化至`lab_reservation`表。

审批流转由`approveReservation`方法处理。方法接收`action`字段（`APPROVE`或`REJECT`）并通过`switch`分支分别执行审批逻辑。驳回分支将预约状态设为`REJECTED`并记录审核意见。通过分支根据预约当前状态判断审批阶段：若当前状态为`PENDING`（教师初审）则更新为`APPROVED`；若当前状态为`APPROVED`（管理员终审）则更新为`PICKUP_PENDING`，同时自动创建`BorrowRecordEntity`写入`lab_borrow_record`表，并将设备状态更新为`RESERVED`。教师角色在审批时还额外校验预约申请人是否属于自己管辖的学生。

预约自动失效通过`SystemTaskScheduler`实现，其中`expireReservationsTask`方法以`@Scheduled(cron = "0 0/30 * * * ?")`注解每30分钟执行一次，调用`SystemTaskService.expirePickupPendingReservations`扫描所有`PICKUP_PENDING`状态且创建时间超过24小时的预约记录，将其状态更新为`EXPIRED`，恢复设备`AVAILABLE`状态，并通过`NotificationService`向申请人发送`RESERVATION_EXPIRED`类型通知。

### 5.5.2 前端设计与实现

前端预约管理页面由`ReservationManagementView.vue`实现。页面通过`computed`属性`canCreate`、`isTeacher`和`isAdmin`识别当前用户角色，动态控制各操作按钮的显示：学生和教师可见"新建预约"按钮，教师和管理员可见"审批"按钮。

新建预约弹窗使用`ElDialog`包裹`ElForm`，包含设备选择下拉框（选项从`listDevices`接口获取）、预约起止时间（使用Element Plus的`ElDateTimePicker`组件，格式为`YYYY-MM-DD HH:mm`）和使用目的输入框。表单提交时调用`createReservation`接口，接口返回409时提示时间冲突。

预约列表通过`ElTable`展示，支持按预约状态筛选。操作列根据当前用户角色和预约状态动态渲染审批、取消等按钮。审批弹窗提供通过/驳回单选，驳回时要求填写审批意见。详情弹窗以只读方式展示完整预约信息，包括申请人、设备、时间段、使用目的和历次审批意见。

## 5.6 借用归还模块设计与实现

借用归还模块负责预约审批通过后的设备实际取用与归还管理，是预约业务在执行阶段的延伸。系统在预约终审通过时自动生成借用记录，并通过定时任务处理逾期场景，形成完整的借还闭环。

### 5.6.1 后端设计与实现

借用归还后端由`BorrowRecordController`和`BorrowRecordService`实现。`listRecords`方法支持按`status`、`userId`和`deviceId`三个维度筛选，同时通过`visibleTo`方法根据当前用户角色控制可见范围：学生只能查看自己的记录，教师可查看其管辖学生的记录，管理员和超级管理员可查看全部记录。

取用操作由`pickup`方法处理。方法校验借用记录当前状态必须为`PICKUP_PENDING`，验证通过后将`status`更新为`BORROWING`，记录`pickupTime`为当前时间，并同步将设备状态更新为`BORROWED`，最后将变更持久化至数据库。

归还操作由`returnDevice`方法处理，接收`deviceCondition`归还状况字段。方法将借用记录`status`更新为`RETURNED`，记录`returnTime`，并根据`deviceCondition`的值判断设备状态：若值为`DAMAGED`，则将设备状态更新为`DAMAGED`；其余情况均恢复为`AVAILABLE`。损坏归还时，系统还自动调用`RepairService.createRepair`创建一条初始状态为`PENDING`的维修申请记录，实现借还流程与维修流程的自动衔接。

逾期处理由`SystemTaskScheduler`中的`overdueBorrowRecordsTask`方法承担，同样以每30分钟的频率执行，调用`SystemTaskService.markOverdueBorrowRecords`扫描所有`BORROWING`状态且`expectedReturnTime`已过期的借用记录，将其`status`更新为`OVERDUE`，并通过`NotificationService`向借用人发送`BORROW_OVERDUE`类型通知。每天上午8时，`reminderSummaryTask`额外执行一次，对即将到期的借用记录生成`ABOUT_TO_EXPIRE_REMINDER`提醒通知。

### 5.6.2 前端设计与实现

前端借用记录展示分布于两个页面。`ProfileCenterView.vue`个人中心页面供学生和教师查看自己的借用记录，支持按状态筛选和分页浏览，展示取用时间、预计归还时间、实际归还时间和设备状况等关键字段。

管理员侧的借用管理功能集成于设备管理页面或独立的借还列表视图中，管理员可对状态为`PICKUP_PENDING`的记录执行取用确认操作，对状态为`BORROWING`的记录执行归还登记操作。归还登记弹窗要求管理员选择设备状况（完好/损坏），选择损坏时系统弹出二次确认提示，告知用户将同时触发维修申请。页面对逾期记录通过`ElTag`的红色标签进行视觉告警。

## 5.7 维修管理模块设计与实现

维修管理模块用于处理设备在使用过程中出现的故障和损坏问题，支持从报修申请提交到维修结果反馈的全流程管理，实现设备损坏信息的规范化记录与处置。

### 5.7.1 后端设计与实现

维修管理后端由`RepairController`和`RepairService`实现。`createRepair`方法在接收报修申请时首先通过`requireRoles`确认申请人为学生角色，随后从`DeviceMapper`查询目标设备是否存在，再调用`ensureNoActiveRepair`方法检查该设备当前是否已有`PENDING`或`PROCESSING`状态的维修记录，若存在则拒绝重复提交。校验通过后创建`RepairEntity`，初始状态设为`PENDING`，并将设备状态更新为`REPAIRING`，以防止设备在维修期间被预约。

`updateRepairStatus`方法供管理员更新维修进度。方法接收目标状态（`PROCESSING`、`COMPLETED`或`UNREPAIRABLE`）和处理说明，根据不同目标状态执行对应的设备状态联动：若更新为`COMPLETED`则将设备状态恢复为`AVAILABLE`；若更新为`UNREPAIRABLE`则将设备状态保持为`DAMAGED`，表示设备无法正常使用。维修状态变更完成后，调用`NotificationService`向申请人发送维修结果通知，通知内容包含处理说明字段，使申请人可及时了解维修结论。

`listRepairs`方法同样通过`visibleTo`进行可见范围控制：学生只能查看自己提交的维修申请，教师可查看其管辖学生提交的申请，管理员可查看全部记录，支持按`status`、`deviceId`和`applicantId`条件筛选分页。

### 5.7.2 前端设计与实现

前端维修管理页面由`RepairManagementView.vue`实现。页面顶部为状态筛选区域，支持按维修状态（待处理、处理中、已完成、无法修复）过滤记录，下方为`ElTable`展示的维修记录列表，包含设备名称、申请人、故障描述、当前状态和申请时间等字段。

管理员在列表中可对待处理或处理中的维修记录执行状态更新操作，点击操作按钮后弹出`ElDialog`，要求管理员选择新状态并填写处理说明。学生可通过列表查看自己提交的维修申请的当前处理进度，但不具备状态更新权限，相关操作按钮对学生角色隐藏。详情弹窗以只读方式展示维修记录的完整信息，包括设备信息、申请人、故障描述、处理说明及状态变更时间。

## 5.8 消息通知模块设计与实现

消息通知模块负责在关键业务节点向用户推送系统消息，贯穿认证、预约、借还和维修等多个业务模块，是提升系统信息可达性的重要支撑组件。

### 5.8.1 后端设计与实现

消息通知后端由`NotificationController`和`NotificationService`实现。通知的生成采用被动调用模式，即由各业务模块在关键节点主动调用`NotificationService.createSystemNotificationIfAbsent`方法触发。该方法首先通过`NotificationMapper.countUnread`查询是否已存在同类型、同关联业务ID的未读通知，若已存在则跳过创建，避免重复推送；若不存在则构建`NotificationEntity`并持久化至`lab_notification`表。触发场景涵盖首次登录（`FIRST_LOGIN_PASSWORD_CHANGE`）、密码重置（`PASSWORD_RESET`）、预约失效（`RESERVATION_EXPIRED`）、借用逾期（`BORROW_OVERDUE`）、即将到期提醒（`ABOUT_TO_EXPIRE_REMINDER`）和逾期提醒（`OVERDUE_REMINDER`）六种类型。

`listMyMessages`方法供用户查询个人消息列表，通过`authService.currentUser()`获取当前用户ID，结合`is_read`状态和`type`类型进行过滤筛选，支持分页返回。`summary`方法返回各类未读消息的数量汇总，前端用于在导航栏渲染消息角标。`confirmMessage`方法在用户确认消息时将`is_read`字段置为`true`，并记录`read_at`时间戳。

定时任务方面，`SystemTaskScheduler`中的`reminderSummaryTask`方法以`@Scheduled(cron = "0 0 8 * * ?")`注解在每天上午8时执行，调用`SystemTaskService.generateReminderSummary`扫描即将到期和已逾期的借用记录，为相关用户批量生成提醒通知。

### 5.8.2 前端设计与实现

前端消息中心页面由`MessageCenterView.vue`实现。页面顶部提供已读/未读状态切换标签和消息类型筛选下拉框，下方以卡片或列表形式展示消息内容，按创建时间倒序排列，最新消息置顶显示。未读消息通过样式差异（加粗标题或高亮背景）与已读消息明显区分，用户可单条或全部标记为已读。

消息角标功能通过在`AdminLayout.vue`布局组件中调用`/api/notifications/summary`接口实现，将未读消息总数展示于顶部导航栏的消息入口旁。该接口在用户登录后初始调用，并在用户访问消息中心页面确认消息后刷新计数，保证角标数量与实际未读消息数量保持一致。

## 5.9 统计分析模块设计与实现

统计分析模块主要用于对系统运行过程中产生的业务数据进行汇总和分析，为管理人员提供设备使用概况和多维度排行数据，辅助实验室管理决策。

### 5.9.1 后端设计与实现

统计分析后端由`StatisticsController`和`StatisticsService`实现。`overview`方法汇总系统当前业务规模的核心指标，通过各Mapper的`selectAll`方法加载全量数据后在内存中统计：设备总量及按状态细分的各类数量、预约总量、借用总量（含逾期数）和维修总量。为减少高频调用时的重复计算开销，`overview`方法引入了`AppCacheService`缓存机制，以`statistics:overview`为键名对结果进行短时缓存，在Redis可用时使用`RedisAppCacheService`，否则降级为`InMemoryAppCacheService`。

热门设备统计由`hotDevices`方法实现，接收`rankScope`（`TOTAL`/`HALF_YEAR`/`MONTH`）和`topN`参数。方法从`BorrowRecordMapper`加载所有借用记录，根据`rankScope`过滤时间范围，通过Java Stream的`Collectors.groupingBy`按`deviceId`聚合统计借用次数，按次数降序排列后取前`topN`条记录，并关联`DeviceMapper`补充设备名称等展示字段。设备损坏统计`deviceDamageStatistics`和用户违规统计`userViolationStatistics`采用类似的聚合分析思路，分别统计各设备的维修申请次数和各用户的逾期及损坏归还次数。

### 5.9.2 前端设计与实现

前端统计分析页面由`StatisticsView.vue`实现，同时作为登录后的默认首页。页面顶部为概览卡片区域，通过`computed`属性`summaryCards`将`overview`接口返回的数据转换为包含标题、数值和描述的卡片列表，使用Element Plus的`ElCard`组件均匀排列展示设备总量、当前可用设备数、借用中设备数、预约总量、借用总量和维修总量等指标。

页面中部为排行榜控制区域，提供`rankScope`（累计/近半年/近一个月）和`topN`（3/5/10）两个筛选选项，均以`ElSelect`下拉框实现，选项变化时自动触发`loadRankings`方法重新拉取三类排行榜数据。排行榜区域以并排的`ElTable`分别展示热门设备榜、设备损坏榜和用户违规榜，支持按数值列排序，帮助管理员快速识别高频使用设备和高风险用户。

## 5.10 前端路由与权限控制实现

为保证不同角色用户只能访问其权限范围内的功能页面，前端采用路由守卫与角色元数据相结合的方式实现访问控制，并通过Pinia状态存储保障登录状态在页面刷新后的持久性。

路由配置集中于`router/index.ts`和`router/access.ts`两个文件。`access.ts`通过`AppRouteMeta`接口为每条路由定义元数据，其中`roles`字段声明允许访问该路由的角色列表，`allowFirstLogin`字段标记首次登录用户是否可访问该路由。`appRouteChildren`数组集中定义所有业务路由，各路由通过`meta`字段关联权限元数据。

路由守卫在`router/index.ts`的`router.beforeEach`钩子中实现，每次路由跳转前执行以下判断：若目标路由`meta.public`为`true`（登录页）且当前用户已登录，则直接重定向至首页；若目标路由要求认证但当前无令牌，则重定向至登录页并携带`redirect`参数；认证通过后调用`authStore.ensureCurrentUser`获取最新用户信息，再通过`canAccessRoute`方法判断当前用户角色是否满足目标路由的`roles`约束，权限不足时跳转至`/403`无权限页面；对于首次登录用户，若目标路由`allowFirstLogin`不为`true`，则同样跳转至`/403`，引导用户完成密码修改。

`getAccessibleMenuItems`函数根据当前用户角色从`appRouteChildren`中筛选出具有`menu:true`标记且角色满足条件的路由，生成侧边栏导航菜单列表，由`AdminLayout.vue`布局组件调用后动态渲染，实现菜单项随角色自动变化的效果。

## 6.5 性能与安全测试

### 6.5.1 性能测试分析

在本地测试环境下，对系统常用接口进行了基础响应时间测量。测试数据量设置为设备约50条、预约记录约200条、借用记录约100条、用户约30名，与典型高校实验室中小规模使用场景相近。在该数量级下，设备列表查询接口的响应时间约为30至60毫秒，用户登录接口约为20至40毫秒，预约记录查询接口约为40至80毫秒，统计概览接口（无缓存情况下）约为80至120毫秒。各核心接口均能在100毫秒内完成响应，满足日常使用场景下的基本性能要求。

### 6.5.2 安全测试分析

安全测试覆盖了以下典型场景：直接访问受保护接口但不携带令牌，系统返回401状态码，拒绝未认证访问；携带格式错误或已过期的令牌访问接口，系统无法在令牌存储中找到对应用户，同样返回401错误；以学生角色令牌访问管理员专属接口（如设备新增、预约审批），系统返回403错误，拒绝低权限访问；以教师角色尝试查看或操作不属于其管辖学生的数据，业务层的`visibleTo`校验生效，返回403错误，越权访问被有效拦截。

密码安全方面，系统对新建账号和密码重置均设置了强制首次修改密码机制，`AuthTokenFilter`确保持有默认密码令牌的用户在修改密码前无法访问任何业务接口。新密码长度不足8位时，`changePassword`方法抛出业务异常并返回明确提示，防止用户设置过短的弱密码。整体来看，系统在身份认证和权限控制方面具备较好的安全保障能力。

## 6.6 测试总结

通过后端集成测试、前端单元测试、功能验收测试以及基础性能与安全测试，本系统在主要业务功能、权限控制、状态流转和异常处理等方面均达到了预期设计目标。系统较好地实现了实验室设备预约管理的核心业务闭环，从预约申请、分级审批、借用确认到归还登记和维修处理，各环节数据流转准确，状态一致性良好。多角色权限控制在角色访问限制和数据可见范围两个维度上均表现稳定，认证机制在正常和异常场景下的响应符合设计规范。在中小规模数据量条件下，系统响应性能满足实际使用需求，基础安全防护能力可覆盖高校实验室信息管理系统的常见安全场景。综合测试结果表明，本系统已基本实现预期功能目标，具备一定的实用价值。

# 第7章 总结与展望

## 7.1 工作总结

本文针对高校实验室设备管理中信息化程度不足、流程分散、数据追溯困难等现实痛点[3][9]，研究并实现了一套基于SSM框架的高校实验室设备预约管理系统。系统采用Spring Boot整合Spring MVC、Spring Security和MyBatis构建后端服务，以Vue 3配合Element Plus、Pinia和Vue Router构建前端管理界面，前后端通过RESTful JSON接口进行数据交互，整体架构清晰、职责分明[17]。

在功能实现方面，系统覆盖了高校实验室设备管理的核心业务需求。**认证模块**基于无状态令牌机制实现了身份认证，结合Spring Security过滤链完成请求级访问控制，并设计了首次登录强制改密机制保障账户安全。**用户管理模块**按照超级管理员、管理员、教师和学生四类角色构建了分层账户管理体系，基于RBAC模型[11]对各角色的操作边界进行严格约束。**设备管理模块**维护了设备从信息录入到全生命周期状态联动的完整流程，支持图片上传和多条件组合检索，实现了设备可用状态的实时准确反映[15]。**预约管理模块**实现了基于申请人角色的自适应分级审批流转（学生申请需经教师初审再由管理员终审、教师申请直接进入终审），通过时间冲突检测防止设备资源重复占用，定时任务自动处理超期失效预约。**借用归还模块**在预约审批通过后自动生成借用记录，通过定时扫描实现逾期状态自动标记和提醒推送，完整记录设备借用全程的数字化轨迹[9]。**维修管理模块**建立了从损坏报告到修复反馈的完整管理闭环，设备损坏归还时可自动触发维修申请，确保损坏信息不被遗漏。**消息通知模块**采用幂等推送机制，在各业务关键节点向相关用户推送通知，覆盖六类业务通知场景。**统计分析模块**提供业务概览卡片和支持三种时间维度的热门设备榜、设备损坏榜、用户违规榜，为管理人员决策提供量化数据参考[7]。

通过后端集成测试、前端单元测试、功能验收测试及性能安全测试的综合验证，各功能模块均能按照设计预期正常运行，业务流程完整性、权限控制有效性和设备状态流转一致性均达到预期标准。系统在中小规模高校实验室应用场景下具备较好的实用性，能够有效替代传统人工登记和纸质审批管理方式，推动实验室设备管理的规范化与信息化进程[2]。

## 7.2 系统创新点

本系统在设计与实现过程中，结合高校实验室管理的实际业务特点，形成了以下几个值得关注的设计亮点。

第一，设计了基于角色差异的自适应审批流转机制。系统并非对所有预约申请采用统一的固定审批链路，而是根据申请人角色动态决定审批起点：学生申请从教师初审开始，教师申请直接进入管理员终审阶段。这一设计在保留多层监管能力的同时，有效减少了不必要的审批层级，使审批流程更贴合实际组织结构。

第二，实现了设备状态的全业务周期联动管理。系统将设备状态（可用、已预约、借出中、维修中、损坏、停用）与预约、借用、归还和维修等各业务操作深度绑定，任一业务节点的状态变化均会实时同步至设备表，确保系统中的设备状态始终准确反映设备的真实可用情况，从根本上解决了信息滞后导致的重复占用问题。

第三，引入了幂等消息推送机制。消息通知模块在创建通知前首先检查是否已存在相同类型且关联相同业务对象的未读通知，避免因定时任务重复执行或业务并发操作导致相同消息被重复推送，在保证消息送达的同时有效控制了通知数量，提升了用户的消息体验。

第四，采用了前端路由元数据驱动的权限控制模式。通过在路由配置的`meta`字段中声明`roles`和`allowFirstLogin`等权限属性，将权限规则与路由定义集中管理，路由守卫和菜单渲染均基于相同的元数据源进行判断，消除了权限规则分散维护带来的不一致风险，使前端访问控制逻辑清晰且易于扩展。

## 7.3 不足与展望

尽管本系统已基本实现高校实验室设备预约管理的核心功能，但受开发周期和应用场景规模的限制，系统在部分方面仍存在一定不足，有待在后续工作中进一步完善。

在技术实现层面，系统当前采用基于内存的令牌存储方案，在单实例部署下运行稳定，但在多节点水平扩展场景下会出现令牌共享问题，无法直接支持分布式部署。此外，设备列表、预约记录等查询当前采用全量加载后内存过滤的方式，随数据量增长性能下降明显，需要引入数据库层面的条件查询和分页支持加以优化。统计分析模块中的概览数据虽已引入缓存机制，但三类排行榜数据每次请求均实时计算，在数据量较大时存在一定性能压力。

在功能完善层面，系统目前不支持通过Excel批量导入设备信息，设备录入效率在数量较多时有限；前端仅提供Web管理端，尚未开发移动端应用，限制了学生和教师在移动场景下的使用便利性；消息通知当前依赖用户主动进入消息中心查看，缺少浏览器推送或邮件等主动触达渠道，对于重要提醒的及时性有一定影响。

针对上述不足，对系统未来的改进方向做出以下展望。一是将令牌存储迁移至Redis集群，解决分布式部署下的会话共享问题，同时利用Redis过期机制自动处理令牌失效，降低存储维护成本。二是将关键查询接口改造为数据库层分页查询，结合适当的索引优化，提升系统在中大规模数据量下的查询性能。三是增加Excel批量导入设备的功能，支持管理员通过标准模板一次性录入大批量设备信息，降低设备初始化的操作成本。四是开发基于微信小程序或移动Web的轻量端应用，使学生和教师能够在移动设备上方便地发起预约申请、查看借用状态和接收消息通知，进一步提升系统的使用便利性。五是集成浏览器Web Push或电子邮件通知渠道，对借用到期、设备逾期等高优先级事件提供主动触达能力，提高关键消息的送达率和处理及时性。

随着高校信息化建设的持续推进，实验室设备管理系统在功能深度和技术架构上均面临更高的要求。本系统的研究与实现为高校实验室信息化管理提供了一套可参考的实践方案，后续将在上述方向持续优化，不断提升系统的稳定性、易用性和可扩展性，以更好地服务于高校实验室的日常管理工作。

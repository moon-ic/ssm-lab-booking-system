# 设备借用后台管理系统 PRD

## 1. 背景与目标

本项目旨在建设一套面向学校或教学场景的设备借用后台管理系统，用于统一管理设备信息、预约申请、借用归还、维修处理、用户账号和权限控制。

系统需要覆盖超级管理员、管理员、教师、学生四类角色，并支持从账号建立、设备预约、审核、借用归还到统计分析的完整闭环。

本 PRD 的目标不仅是描述产品功能，还要为下一步 API 文档设计提供稳定输入，帮助后续明确核心资源、状态流转、权限边界和业务规则。

## 2. 问题定义

当前设备借用管理通常存在以下问题：

- 设备信息分散，难以统一维护
- 预约、审核、借用、归还流程缺乏系统化管理
- 不同角色的权限边界不清晰
- 设备维修、损坏、违规记录缺少统一追踪
- 后续统计分析缺乏结构化数据支撑

需要通过一套后台管理系统解决以上问题，提升设备流转效率、降低管理成本，并为后续功能扩展和接口标准化提供基础。

## 3. 产品目标

- 建立统一的设备借用管理后台
- 明确不同角色的权限范围和操作边界
- 规范设备预约、审核、借用、归还、维修流程
- 支持个人中心与信用分、借阅记录管理
- 提供可用于运营与管理决策的数据统计能力
- 为后续 API 文档和前后端实现提供清晰业务基础

## 4. 用户角色

### 4.1 超级管理员

负责系统级管理，并默认拥有管理员的全部权限。重点包括：

- 创建和管理管理员账号
- 管理角色和权限体系
- 配置菜单/路由权限
- 查看系统级统计和运行情况
- 执行管理员可执行的全部业务操作

### 4.2 管理员

负责日常业务管理，重点包括：

- 创建教师账号
- 管理设备信息
- 审核预约申请
- 处理设备维修状态
- 查看统计分析结果

### 4.3 教师

负责教学场景下的学生与预约管理，重点包括：

- 创建和删除学生账号
- 审核设备预约申请
- 查看学生借用情况

### 4.4 学生

作为设备借用的主要申请者，重点包括：

- 登录系统
- 查看个人信息
- 查看设备列表并发起预约
- 确认取设备
- 归还设备
- 提交维修申请

## 5. 功能范围

### 5.1 范围内

- 登录与账号管理
- 权限与角色管理
- 用户个人中心
- 设备管理
- 预约与审核
- 借用与归还
- 到期提醒
- 设备维修
- 统计分析
- 图标选择或图标库功能

### 5.2 非目标

本阶段默认不包含以下内容：

- 第三方统一身份认证接入
- 支付、押金、财务结算能力
- 复杂排班系统
- 物联网设备自动识别或自动开锁
- 消息中心之外的外部短信、邮件、微信通知集成

## 6. 核心业务流程

### 6.1 账号创建流程

1. 超级管理员创建和管理管理员账号。
2. 管理员创建教师账号。
3. 教师创建学生账号。
4. 教师可删除学生账号。
5. 新建账号默认密码为 `123456`。
6. 用户首次登录后可修改密码。

### 6.2 设备预约流程

1. 用户查看设备列表。
2. 用户按条件检索设备。
3. 用户选择可预约设备并填写预约时间、用途。
4. 系统提交预约申请。
5. 教师或管理员进行审核。
6. 审核通过后，预约进入待取设备状态。
7. 用户在线下确认取设备或在系统中确认取设备。

### 6.3 借用归还流程

1. 预约审核通过后，用户确认取设备。
2. 设备状态更新为借用中。
3. 到期前系统进行提醒，例如提前 3 天提醒。
4. 用户归还设备。
5. 设备状态更新为可用或进入待维修状态。
6. 借阅记录归档，可在个人中心查看。

### 6.4 维修流程

1. 用户在发现设备异常后提交维修申请。
2. 管理员查看维修申请。
3. 管理员更新维修状态。
4. 维修完成后设备可重新投入使用，或标记为不可用。

## 7. 核心功能需求

### 7.1 登录与账号管理

- 学生使用学号登录
- 教师使用工号登录
- 管理员和超级管理员使用系统分配账号登录
- 默认密码为 `123456`
- 支持修改密码
- 管理员账号由超级管理员创建和管理
- 教师账号由管理员创建
- 学生账号由教师创建和删除

补充规则：

- 不同角色的账号创建权限必须严格隔离
- 默认密码策略需要支持后续扩展，例如首次登录强制修改密码
- 应保留账号启用、禁用能力，当前若未实现需列为待确认项

### 7.2 权限与角色管理

- 支持角色管理
- 支持新增角色
- 支持为角色分配权限
- 支持菜单/路由权限配置
- 不同角色访问不同功能模块

补充规则：

- 角色权限需要至少覆盖菜单可见性和功能操作权限
- 角色与用户之间应支持分配关系
- 菜单权限和功能权限是否完全一致，需要在设计阶段继续确认

### 7.3 用户个人中心

- 查看个人信息
- 个人信息至少包括姓名、学号或工号、信用分
- 查看借阅记录
- 借阅记录需区分已归还和未归还
- 支持修改密码

补充规则：

- 信用分变更来源及扣减规则当前未明确，需列为待确认
- 借阅记录需要保留设备、借用时间、归还时间、状态等基础信息

### 7.4 设备管理

- 查看设备列表
- 设备状态至少区分未预约和已预约
- 支持设备检索
- 支持设备预约
- 预约时填写时间和用途
- 管理员可上传或新增设备

补充规则：

- 设备应具备唯一标识
- 设备列表后续需要支持更多状态，不应只局限于未预约和已预约
- 设备上传是批量导入还是附件上传，当前需待确认
- 检索条件至少应支持设备名称、分类、状态，其他条件待确认

### 7.5 预约与审核

- 用户提交设备预约申请
- 教师和管理员可审核预约

补充规则：

- 同一预约是否允许多级审核，当前未明确，默认任一有权限角色可完成审核
- 审核结果至少包括通过、驳回
- 驳回时是否必须填写原因，建议设为必填
- 审核通过后应锁定对应时间段内设备资源

### 7.6 借用与归还流程

- 用户确认取设备
- 用户归还设备
- 系统提供即将到期和已过期提醒

补充规则：

- 到期提醒默认支持提前 3 天提醒
- 已过期后需有明显状态标记
- 是否允许管理员代确认归还，当前需待确认
- 超期是否影响信用分，当前需待确认

### 7.7 设备维修

- 用户可提交设备维修申请
- 管理员更新维修状态

补充规则：

- 维修申请应关联具体设备和申请人
- 维修状态建议至少包括待处理、处理中、已完成、无法修复
- 设备维修期间是否允许继续预约，默认不允许

### 7.8 统计分析

- 热门设备排行
- 设备损坏情况统计
- 用户违规记录统计

补充规则：

- 热门设备排行应基于借用次数、预约次数或其他指标，具体口径待确认
- 损坏情况统计需依赖维修记录和设备状态
- 用户违规记录统计需明确违规定义，例如超期未还、损坏未报、违规借用

### 7.9 图标选择或图标库功能

- 提供图标选择能力
- 支持图标库功能
- 主要用于界面配置

补充规则：

- 图标功能主要服务于菜单、模块或界面配置
- 图标来源是内置图标库还是外部图标库，当前待确认

### 9.2 管理员

- 管理设备

### 9.3 教师

## 10. 关键业务规则

- 学生登录标识为学号，教师登录标识为工号
- 新建账号默认密码统一为 `123456`
- 账号创建权限按角色逐级下放，不允许越权创建
- 超级管理员默认拥有管理员全部权限
- 设备预约必须填写预约时间和用途
- 教师和管理员都有预约审核权限
- 借用前必须经过预约审核通过
- 设备在已预约、借用中、维修中状态下默认不可重复预约
- 系统支持到期前提醒和已过期提醒
- 维修申请提交后，设备处理状态应可追踪
- 借阅记录需支持按已归还和未归还分类查询

## 11. 边界情况与异常场景

- 同一时间段重复预约同一设备时如何处理
- 审核通过后用户未取设备时如何处理
- 用户逾期未归还时如何处理
- 归还时设备损坏时如何处理
- 维修中设备是否允许查看但不可预约
- 已被禁用的账号是否保留历史借阅与审批数据
- 教师删除学生账号前，若该学生存在未归还设备时如何处理

默认建议：

- 冲突预约直接拒绝
- 未取设备的预约应支持过期失效
- 历史业务数据不因账号删除而物理删除

## 12. 验收标准

- 四类角色可完成各自权限内的系统操作
- 账号创建链路清晰且不可越权
- 设备可完成从展示、预约、审核、借用、归还到维修的完整业务流转
- 用户可在个人中心查看个人信息和借阅记录
- 管理端可查看统计分析数据
- 权限控制可限制不同角色访问不同模块
- PRD 中的信息足以继续产出 API 文档

## 13. 待确认项

- 管理员和超级管理员的登录标识规则
- 是否要求首次登录强制修改密码
- 用户账号是否支持启用、禁用、重置密码
- 信用分初始值、扣减规则、恢复规则
- 预约审核是否支持多级审核
- 审核通过后设备锁定的具体规则
- 到期提醒和过期提醒的通知方式
- 是否允许管理员代用户执行取用、归还确认
- 超期是否自动影响信用分或违规记录
- 设备上传的具体形式是单条新增、批量导入还是文件导入
- 图标库是内置维护还是接入外部资源
- 统计分析的时间维度和统计口径
